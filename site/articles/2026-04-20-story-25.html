webpack

오래된 빌드 도구를 유지한다는 것은 멈춰 있는 선택이 아니라, 생태계 전체의 최소 런타임 기준을 함께 올리는 작업입니다.

webpack 5.106에서 source phase Wasm import, create-webpack-app, custom JavaScript parser 지원, webpack-cli 7 및 webpack-dev-middleware 8 생태계 업데이트가 안내됐습니다.

2026-04-20 | TOOL

서비스 변화 요약

  • 업데이트webpack 5.106에서 source phase Wasm import, create-webpack-app, custom JavaScript parser 지원, webpack-cli 7 및 webpack-dev-middleware 8 생태계 업데이트가 안내됐습니다.
  • 서비스 맥락webpack은 레거시 번들러가 아니라 대형 프로젝트와 플러그인 생태계를 유지하면서 modern runtime, TypeScript config, 성능 실험을 흡수하는 방향으로 진화하고 있습니다.
  • 변경 전신규 webpack 프로젝트 scaffold는 webpack-cli init 흐름에 묶여 있었고, parser 교체나 Wasm import 실험은 프로젝트별 설정 부담이 컸습니다.
  • 변경 후npx create-webpack-app로 프로젝트, custom loader, plugin boilerplate를 만들 수 있고, custom parser를 특정 JavaScript 파일에 적용해 Oxc 같은 대체 parser 실험을 할 수 있습니다.
  • 주요 변화source phase Wasm import, standalone create-webpack-app, custom JavaScript parser, webpack-cli 7, webpack-dev-middleware 8
  • Node 기준webpack-cli 7, webpack-dev-middleware 8, 주요 minimizer/compression plugin 계열이 최소 Node.js 20.9.0 기준으로 정렬됐습니다.

인사이트

오래된 빌드 도구를 유지한다는 것은 멈춰 있는 선택이 아니라, 생태계 전체의 최소 런타임 기준을 함께 올리는 작업입니다.

왜 지금 이 업데이트인가

Vite와 Rolldown처럼 Rust 기반 신규 도구가 주목받는 상황에서도 webpack은 여전히 많은 기업 프로젝트의 표준 빌드 체인입니다. webpack 5.106은 큰 headline보다 운영 중인 생태계를 현대화하는 업데이트에 가깝습니다.

특히 create-webpack-app 분리, TypeScript config 로딩 개선, custom parser 실험, Node 20.9.0 기준 정렬은 webpack을 새 프로젝트와 기존 프로젝트 양쪽에서 유지 가능하게 만드는 장치입니다.

프론트엔드 개발 전문가 관점

webpack을 쓰는 팀은 "언제 Vite로 갈 것인가"만 물어서는 부족합니다. 현재 webpack 기반 프로젝트가 Node 버전, CLI, dev middleware, minimizer plugin, config loader 기준을 따라가고 있는지 먼저 봐야 합니다.

custom parser 지원은 Oxc 같은 빠른 parser를 실험할 여지를 열지만, production 기본값으로 바로 넣기보다 benchmark branch나 개발 환경에서 호환성을 확인하는 방식이 맞습니다.

실무에 어떻게 적용할 수 있을까

  • 기존 webpack 프로젝트는 CSS loader, asset module, SVG loader, image minimizer가 Node 20.9.0 기준에서 그대로 동작하는지 먼저 확인합니다.
  • create-webpack-app은 내부 화면 구현 스타터나 캠페인 페이지 템플릿을 표준화할 때 사용합니다.
  • custom parser와 Oxc 실험은 production 적용 전 HTML/CSS/JS 번들 크기, sourcemap, legacy browser 지원 결과를 비교하는 별도 branch에서 검증합니다.

같이 보면 좋은 기술

  • webpack-cli 7: scaffold와 config 실행 흐름이 바뀌므로 기존 프로젝트 생성/빌드 스크립트와 함께 확인해야 합니다.
  • webpack-dev-middleware 8: dev server 통합을 직접 구성한 프로젝트는 middleware 버전 기준이 빌드 도구와 맞아야 합니다.
  • Source phase Wasm import: Wasm 모듈을 JavaScript 번들 안에서 다루는 방식이 바뀌는지 확인할 수 있습니다.
  • Custom JavaScript parser: Oxc 같은 대체 parser 실험을 webpack 안에서 제한적으로 검증할 수 있습니다.
  • Node 20.9.0 기준: CLI와 플러그인 생태계의 최소 런타임이 올라가므로 CI와 로컬 개발 환경을 함께 맞춰야 합니다.
