기술 변화 요약

  • 업데이트ESLint v10.2.1 패치가 공개됐고, 직전 v10.2.0에서는 language-aware rules와 Temporal 인식이 추가됐습니다.
  • 기술 맥락ESLint가 JavaScript 전용 lint를 넘어 언어별 rule 적용 범위를 명시하는 방향으로 확장되고 있습니다.
  • 변경 전rule이 어떤 언어/파서 조합을 지원하는지 명확하지 않으면 TypeScript, Markdown, JSON 등 다언어 lint 구성에서 오동작 위험이 있었습니다.
  • 변경 후custom rule의 meta.languages로 지원 언어를 선언할 수 있고, 지원되지 않는 언어에 rule이 켜지면 런타임 오류로 드러납니다.
  • 확인할 영향주요 변화 meta.languages, Temporal global 인식, no-obj-calls Temporal 처리 / 최신 패치 v10.2.1에서 code path analysis와 no-async-promise-executor false positive 등 버그 수정

인사이트

프론트엔드 품질 도구는 이제 파일 확장자를 넘어서 언어별 의미를 구분하는 단계로 가고 있습니다.

왜 지금 이 업데이트인가

현대 프론트엔드 저장소는 JavaScript와 TypeScript만으로 구성되지 않습니다. Markdown, JSON, CSS-in-JS, MDX, 설정 파일이 한 모노레포 안에서 함께 lint됩니다. ESLint가 language-aware rule을 도입한 것은 이 복잡한 구성에서 rule 적용 범위를 더 명확히 하려는 변화입니다.

Temporal 지원도 같은 흐름입니다. 표준 API가 추가될 때 lint 도구가 이를 모르면 no-undef 같은 기본 rule이 잘못된 경고를 만들 수 있습니다.

프론트엔드 개발 전문가 관점

팀은 ESLint v10 계열을 단순 버전 업데이트로 보면 안 됩니다. 사내 custom rule이나 플러그인이 있다면 어떤 language option에서 동작하는지 명시해야 하고, TypeScript/Markdown/JSON lint 범위가 섞인 구성에서는 rule 충돌을 점검해야 합니다.

실무에 어떻게 적용할 수 있을까

  • 화면 구현 산출물에 섞이는 JS, TS, MDX, JSON, CSS-in-JS 파일을 패턴별로 분리해 lint rule 적용 범위를 정합니다.
  • 접근성, alt, button type, aria 속성처럼 마크업 품질에 직접 연결되는 rule은 PR에서 반드시 실패하도록 둡니다.
  • 문서나 JSON에 JavaScript rule이 잘못 적용되는 경우를 줄이기 위해 flat config를 파일 타입별로 나눕니다.

같이 보면 좋은 기술

  • ESLint flat config: 파일 패턴별 rule 적용 범위를 명시해야 language-aware rules의 효과가 납니다.
  • @typescript-eslint: TypeScript 파서와 rule이 ESLint v10의 언어 구분과 충돌하지 않는지 확인해야 합니다.
  • Markdown/JSON lint plugins: 다언어 저장소에서 JavaScript rule이 문서나 설정 파일에 잘못 적용되는 문제를 줄일 수 있습니다.
  • custom rule meta.languages: 사내 rule이 어떤 언어에서 동작하는지 선언해야 예기치 않은 런타임 오류를 막을 수 있습니다.
  • monorepo lint pipeline: 패키지별 lint 범위와 CI 실행 순서가 rule 적용 범위와 맞는지 같이 점검해야 합니다.
