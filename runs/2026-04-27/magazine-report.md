- 변경 전: Safari에서만 달라지는 폼, 스크롤, CSS, JavaScript 동작은 배포 후 발견되기 쉬웠습니다.
- 변경 후: TP 릴리즈를 통해 다음 QA 항목을 미리 잡을 수 있습니다.
- 확인할 영향: CTTD 프로젝트에 영향 있는 CSS, HTML, Forms, JavaScript 항목만 골라 테스트해야 합니다.

##### 매거진 인사이트

###### 왜 지금 이 업데이트인가

Safari는 커머스와 콘텐츠 서비스의 실제 모바일 사용 환경에서 여전히 중요합니다. Technology Preview 릴리즈는 새 기능보다 다음 배포에서 깨질 수 있는 컴포넌트를 미리 보는 자료입니다.

###### 구현 관점

릴리즈 노트는 모두 읽기보다 우리 서비스의 폼, 모달, 스크롤, 이미지, SVG, 접근성 컴포넌트와 연결되는 항목만 추려야 합니다.

###### 실무에 어떻게 적용할 수 있을까

- iOS/Safari 의존도가 높은 결제, 로그인, 상품상세 화면을 우선 테스트합니다.
- select, dialog, sticky, scroll 관련 변경을 컴포넌트별로 확인합니다.
- Safari TP에서 발견한 차이는 정식 Safari QA 체크리스트로 옮깁니다.

###### 같이 보면 좋은 기술

- WebKit Release Notes: Safari 변경 사항의 1차 기준입니다.
- BrowserStack: Safari 환경별 회귀 테스트에 활용할 수 있습니다.
- CSS Support Matrix: 기능별 브라우저 지원 상태를 확인합니다.

- 업데이트: Chrome Developers가 web app manifest의 localization support를 소개했습니다.
- 기술 맥락: PWA 설치 화면의 이름, 설명, 아이콘 맥락은 사용자가 앱을 신뢰하는 첫 진입 정보입니다.
- 변경 전: 다국어 서비스는 설치형 웹 경험에서 언어별 표현을 세밀하게 나누기 어려웠습니다.
- 변경 후: manifest 안에서 지역과 언어에 맞는 이름과 설명을 제공하는 흐름이 강화됩니다.
- 확인할 영향: 글로벌 커머스나 다국어 서비스의 설치 경험 QA 항목으로 볼 수 있습니다.

##### 매거진 인사이트

###### 왜 지금 이 업데이트인가

웹앱 설치 경험은 작지만 사용자가 서비스를 앱처럼 인식하는 중요한 순간입니다. 다국어 서비스에서 설치 이름과 설명이 어색하면 신뢰가 낮아질 수 있습니다.

###### 구현 관점

manifest localization은 번역 파일 하나를 추가하는 문제가 아닙니다. 각 언어의 브랜드명, 짧은 설명, 아이콘, 스토어성 문구가 실제 설치 화면에서 자연스러운지 확인해야 합니다.

###### 실무에 어떻게 적용할 수 있을까

- PWA manifest의 name, short_name, description을 언어별로 점검합니다.
- 설치 프롬프트와 홈 화면 표시 이름이 브랜드 정책과 맞는지 확인합니다.
- Chrome 외 브라우저 지원 상태를 별도로 검토합니다.

###### 같이 보면 좋은 기술

- Web App Manifest: 설치형 웹 경험의 기본 메타데이터입니다.
- i18n Routing: 언어별 URL과 표시 정보를 연결합니다.
- Lighthouse PWA Audit: 설치 가능성과 manifest 품질을 점검합니다.

###### 실무에 어떻게 적용할 수 있을까

- UI 테스트 자동화 도구는 기존 E2E 전략과 중복되는지 확인합니다.
- 미디어/그래픽 UI 도구는 접근성, 성능, 브라우저 지원을 함께 검토합니다.
