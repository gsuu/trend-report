> 선물 UX는 상품을 많이 보여주는 것보다, 누가 누구에게 어떤 상황에서 주는지 먼저 좁혀줄 때 강해집니다.

###### 왜 지금 이 업데이트인가

W컨셉의 키즈 원더랜드는 단순 어린이날 할인전보다 선물 구매의 탐색 방식을 보여주는 사례에 가깝습니다. 키즈 상품은 실제 사용자와 구매자가 다르고, 구매자는 아이의 취향과 부모의 기준, 선물 예산을 동시에 고려해야 합니다.

###### 설계 관점

###### 사용자는 무엇을 덜 해도 될까

사용자는 브랜드를 하나씩 검색하거나 어린이날 선물 기준을 따로 세우지 않아도 됩니다. 서비스가 시즌, 연령, 취향, 혜택을 먼저 묶어주면 사용자는 “무엇을 사야 하는가”보다 “이 아이에게 맞는가”를 판단하면 됩니다.

###### 점검 질문

시즌 기획전에서는 “몇 퍼센트 할인할 것인가?”보다 “사용자의 구매 목적을 어떤 기준으로 좁혀줄 것인가?”를 먼저 물어야 합니다. 특히 선물 카테고리는 가격 혜택보다 추천 기준과 실패 확률을 낮추는 화면 구조가 전환에 더 직접적입니다.

###### 왜 지금 이 업데이트인가

직매입과 MFS 풀필먼트는 구매 이후의 불안을 줄입니다. 특히 뷰티는 색상, 제형, 향, 피부 반응처럼 화면만으로 판단하기 어려운 요소가 많기 때문에 온라인 상품 상세에서 배송·교환·반품 조건을 명확히 보여주는 전략이 설득력을 가집니다.

앱 화면에서는 단순히 “빠른 배송” 배지를 붙이는 것으로 충분하지 않습니다. 상품 상세에서 오늘 발송 가능 여부, 주말 수령 가능 지역, 바로교환 조건, 반품/환불 기준이 같은 판단 흐름 안에 보여야 합니다.

###### 사용자는 무엇을 덜 해도 될까

사용자는 브랜드가 낯설다는 이유만으로 배송과 교환 조건을 따로 확인하지 않아도 됩니다. 서비스가 물류와 교환 정책을 전면에 보여주면, 비교 기준은 브랜드 인지도에서 제품 적합성과 구매 편의성으로 이동합니다.

###### 점검 질문

뷰티·패션 서비스에서는 “개인화 추천을 어떻게 할까?”와 함께 “추천된 상품을 안심하고 살 근거가 화면에 있는가?”를 물어야 합니다. 추천 UX와 운영 UX가 분리되어 있으면 사용자는 발견은 하되 구매는 미루게 됩니다.

##### 서비스 변화 요약

- 업데이트: 카카오스타일이 운영하는 지그재그가 `임부복`을 기존 플러스 사이즈 카테고리에서 분리해 별도 독립 카테고리로 신설했습니다.
- 서비스 맥락: 지그재그는 2030 여성의 생애주기 변화에 맞춰 임부복, 수유복, 수영복/래시가드, 이너웨어/케어용품 등 세부 탐색 범위를 확장했습니다.
- 변경 전: 임부복은 플러스 사이즈 안에 묶여 있어 사용자가 상황별 상품을 직접 검색하고 비교해야 했습니다.
- 변경 후: 임신·출산 전후 상황에 맞는 의류와 케어 상품을 독립 카테고리 안에서 탐색할 수 있고, 월 1회 임부복 연합전과 직진배송 연계까지 온라인 구매 흐름으로 연결됩니다.
- 수치·팩트: 2025년 임부복 상품 거래액은 2024년 대비 62% 증가했고, 2026년 1월 1일부터 4월 13일까지 임산부 블라우스 275%, 임산부 브래지어 291%, 임산부 수영복 2549% 등 세부 품목 거래액이 전년 동기 대비 급증했습니다. 임부복 연합전 참여 스토어 평균 거래액도 전년 동기 대비 167% 증가했습니다.

##### 매거진 인사이트

> 좋은 온라인 카테고리 개편은 상품군을 하나 더 늘리는 일이 아니라, 사용자의 상황 언어를 탐색 구조로 바꾸는 일입니다.

###### 왜 지금 이 업데이트인가

지그재그의 임부복 카테고리 신설은 온라인 패션 플랫폼이 고객의 나이보다 생활 단계와 구매 상황을 더 세밀하게 보기 시작했다는 신호입니다. 임부복은 단순히 사이즈가 다른 옷이 아니라 출근, 휴가, 하객룩, 수유, 이너웨어처럼 상황별 판단 기준이 다른 상품군입니다.

###### 설계 관점

온라인 서비스에서는 `임부복`이라는 상위 카테고리만 만드는 것으로 충분하지 않습니다. 사용자는 몸의 변화, 계절, 외출 상황, 배송 속도, 리뷰 신뢰도를 함께 봅니다. 따라서 카테고리 홈에서는 상황별 큐레이션, 빠른 배송 가능 여부, 리뷰가 많은 스토어, 세부 품목 진입점을 함께 보여주는 구조가 중요합니다.

###### 사용자는 무엇을 덜 해도 될까

사용자는 플러스 사이즈, 원피스, 이너웨어, 수영복을 따로 검색하며 임신 중 착용 가능 여부를 추측하지 않아도 됩니다. 서비스가 생애주기와 상황을 먼저 묶어주면 사용자는 자신의 필요에 맞는 상품군으로 바로 진입할 수 있습니다.

온라인 카테고리 개편에서는 “상품군을 추가할 것인가?”보다 “사용자가 어떤 상황 문장으로 상품을 찾고 있는가?”를 먼저 물어야 합니다. 검색어와 거래액이 반복해서 커지는 상품군은 별도 카테고리, 기획전, 배송/리뷰 필터를 함께 설계할 후보입니다.

###### 구현 관점

디자인 시스템을 운영하는 팀이라면 Safari 안정 버전에 들어오기 전부터 테스트 케이스를 준비해야 합니다. 특히 `appearance: base-select`를 검토하는 팀은 스타일링 자유도와 보조기기 호환성을 함께 확인해야 하고, 모달 컴포넌트는 `<dialog closedby>` 지원 흐름을 지켜볼 필요가 있습니다.

###### 실무에 어떻게 적용할 수 있을까

- 커머스 옵션 셀렉트와 필터 컴포넌트에서 VoiceOver 포커스, 라벨, onchange 이벤트를 Safari TP로 회귀 테스트합니다.
- 모달, 바텀시트, 확인창 컴포넌트는 `<dialog>` 기반 전환 가능성과 닫힘 정책을 별도 테스트 케이스로 분리합니다.
- iframe을 쓰는 결제, 외부 리뷰, 라이브커머스 영역은 `prefers-color-scheme` 다크 모드 매칭을 확인합니다.
- IndexedDB를 쓰는 최근 본 상품, 임시 저장, 오프라인 캐시 기능은 `getAllRecords()` 지원이 안정화될 때 데이터 읽기 코드를 단순화할 수 있는지 검토합니다.

###### 같이 보면 좋은 기술

- `appearance: base-select`: 네이티브 select 스타일링과 접근성 균형을 다시 점검할 때 필요합니다.
- `<dialog closedby>`: 모달 닫힘 정책을 명시적으로 제어하는 흐름과 연결됩니다.
- CSS `attr()` Values Level 5: 속성 기반 스타일링을 더 넓게 사용할 가능성을 열어줍니다.
- IndexedDB `getAllRecords()`: 로컬 데이터 조회 코드를 줄일 수 있는 Web API 변화입니다.
