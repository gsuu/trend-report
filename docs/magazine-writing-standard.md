# 매거진 원고 출력 기준

이 문서는 `magazine-report.md`에 들어가는 최종 원고의 출력 형식과 세부 작성 기준만 정의합니다. 후보 선별은 `service-digest-agent-prompt.md`, `design-digest-agent-prompt.md`, `dev-digest-agent-prompt.md`를 따르고, 공통 문체와 제목·요약·태그 원칙은 `editorial-style-guide.md`를 따릅니다.

이 문서는 단일 진실(single source of truth)입니다. `templates/weekly-report.md`, `scripts/collect/prompts.mjs`의 편집 프롬프트, `AGENTS.md`의 Magazine Summary Standard는 모두 이 문서를 기준으로 동기화됩니다. 이 문서가 정한 섹션 구조와 소제목 외에는 **즉흥적인 소제목을 만들지 않습니다**. 4월 28일 회차에서 발견된 `이 레퍼런스에서 먼저 볼 장면`, `검수 포인트` 같은 변형 소제목은 이번 회차부터 폐기합니다.

## 자체 완결 요약 원칙

요약은 그 자체로 글의 메시지가 보이게 씁니다. 독자가 원문을 열지 않아도, 메타데이터의 `요약:` 한 줄과 본문 `##### 요약` 불릿만으로 **누가 무엇을 했고, 무엇이 핵심 변화/주장인지, 왜 이 글을 봐야 하는지**가 드러나야 합니다.

자체 완결 점검 기준:

- 메타 `요약:` 1-2문장은 **주체 + 무엇을 + 어떻게/왜**가 한 문장 안에 들어갑니다. "~를 다룹니다", "~를 소개합니다", "~에 대한 글입니다" 같은 표지 문장으로 끝나면 실패입니다.
- 본문 `##### 요약`의 첫 불릿은 **글 전체의 결론 한 줄**입니다. 이후 불릿은 결론을 뒷받침하는 구체 사실(기능명, 출시 범위, 수치, 화면 변화, 작성자 주장의 근거)을 담습니다.
- 불릿 개수는 제한하지 않습니다. 원문을 대표하는 의미 있는 포인트를 필요한 만큼 담되, 의미 없는 채움 불릿은 넣지 않습니다. 원문 밀도가 높으면 5개, 7개도 됩니다.
- "원문은 ... 다룹니다", "기사에서는 ... 언급됩니다" 같은 메타 표현으로 불릿을 시작하지 않습니다. 원문 안의 사실로 바로 들어갑니다.
- 요약만 읽었을 때 **카테고리·출처 유형·우리 서비스에 던질 질문**까지 자연스럽게 그려져야 통과입니다.

좋은 예 (Service / 당근 AI 후기):

```
요약: 당근이 동네 가게 후기 작성 화면에 음성 입력 기반 AI 인터뷰 기능을 출시해, 사용자가 한 줄도 쓰지 않고도 후기를 남길 수 있게 했습니다. 베타 기간 후기 품질 지표가 40% 올랐다고 공개했습니다.

##### 요약
- 당근은 후기 작성 화면을 빈 입력창에서 음성 답변 기반 인터뷰로 바꿔, 첫 한 줄을 떠올리지 못해 작성을 포기하는 흐름을 끊었습니다.
- AI는 분위기·서비스 특징·이용 메뉴를 차례로 묻고, 음성 답변을 모아 후기 초안으로 정리합니다.
- 1월 베타에서 재이용률 43%, 후기 품질 지표 40% 상승이 공개됐고, 4월 전 업종으로 확대됐습니다.
```

나쁜 예 (피해야 할 패턴):

```
요약: 당근이 새로운 AI 후기 기능을 소개했습니다. 사용자에게 도움이 되는 기능입니다.
- 당근의 보도자료가 AI 후기 기능을 다룹니다.
- 기사에서는 음성 입력이 언급됩니다.
- 베타 결과도 함께 소개됐습니다.
```

이 규칙은 Service, Design, DEV 모두에 동일하게 적용합니다. DEV는 추가로 기술명·API·버전·브라우저 조건 중 하나 이상이 첫 두 불릿 안에 들어가야 합니다.

## 공통 메타데이터

모든 Service, Design, DEV 항목은 아래 메타데이터를 유지합니다.

```markdown
- 날짜: YYYY-MM-DD
- 태그: 원문 중심 태그 2-4개
- 국가: KR|GLOBAL
- 카테고리: category_key
- 직무 태그: 웹서비스기획|웹디자인|웹DEV
- 출처 유형: news|release_note|blog_opinion|reference|guide|research
- 출처: 출처명
- 출처 URL: 최종 기준 원문 URL
- 이미지: 원문 대표 이미지 URL
- 이미지 설명: 이미지 출처와 성격
- 요약: 카드와 상세 상단에 노출되는 1-2문장
```

묶음 DEV 항목처럼 보조 원문이 있으면 `보조 출처 1`, `보조 출처 2`, `관련 링크 1` 형식으로 이어서 씁니다.

## 잠금 소제목 셋 (Locked sub-headings)

각 영역의 인사이트 본문 소제목은 아래 4개로 고정합니다. 다른 표현으로 대체하거나 영역 사이에서 섞지 않습니다.

| 영역 | 본문 섹션 | 인사이트 소제목 (H6, 4개 고정) |
|------|-----------|--------------------------------|
| Service | `##### 요약` → `##### 매거진 인사이트` | 왜 지금 이 업데이트인가 / 사용자는 무엇을 덜 해도 될까 / 설계 관점 / 점검 질문 |
| Design  | `##### 요약` → `##### 디자인 인사이트` | 왜 참고할 만한가 / 어디에 적용할 수 있을까 / 디자인 관점 / 점검 질문 |
| DEV     | `##### 용어 설명`(옵션) → `##### 요약` → `##### 매거진 인사이트` | 왜 지금 이 업데이트인가 / 구현 관점 / 실무에 어떻게 적용할 수 있을까 / 같이 보면 좋은 기술(옵션) |

소제목 추가 금지 항목: `이 레퍼런스에서 먼저 볼 장면`, `적용하기 좋은 화면`, `검수 포인트`, `이번 업데이트에서 바뀐 흐름`, `실제 작업 흐름에서 볼 지점`, `도입 전 검수 포인트`, `가이드가 먼저 보는 문제`, `화면에 옮길 기준`, `체크리스트로 바꿀 항목` 등. 이 표의 4개 외에는 모두 위 4개 중 하나로 흡수해 씁니다.

## SERVICE 출력 형식

SERVICE 글은 서비스 화면, 정책, 전환 흐름, 운영 흐름을 설명합니다. `요약`은 원문 전체 핵심을 불릿 리스트로 정리하고, `매거진 인사이트`는 우리 서비스에 적용할 판단 기준으로 바꿉니다. 각 회차는 기사 나열로 끝내지 말고 `이번 주 모니터링 범위`, `채택한 TOP 항목`, `수집했지만 제외한 것`의 운영 맥락이 함께 읽히게 정리합니다.

회차 앞머리에 붙는 운영 메모는 아래 흐름을 권장합니다.

```markdown
### 이번 주 모니터링 범위

- 중점 관심사 축:
- 확인 플랫폼:
- 이번 주 포인트:

### 채택한 TOP 항목
```

```markdown
##### 요약

- 업데이트: 원문의 기능명, 출시 범위, 달라진 화면이나 정책이 드러나도록 2줄 정도로 씁니다.
- 서비스 맥락:
- 변경 전:
- 변경 후:
- 수치·팩트:

##### 매거진 인사이트

> 이 업데이트의 의미를 압축한 한 문장

###### 왜 지금 이 업데이트인가

###### 사용자는 무엇을 덜 해도 될까

###### 설계 관점

###### 점검 질문
```

SERVICE 작성 기준:

- `요약`은 본문 전체 핵심을 불릿 리스트로 정리합니다. 화면, 정책, 조건, 전환 흐름, 운영 변화처럼 원문에서 확인한 내용을 필요한 만큼 담습니다.
- 각 SERVICE 항목은 `무엇이 바뀌었는가`, `사용자 행동이 어떻게 달라지는가`, `CTTD 클라이언트 질문으로 어떻게 바꿀 수 있는가`까지 이어져야 합니다.

- `업데이트`는 제목 복사가 아니라 원문 핵심을 2줄 정도로 풀어 씁니다.
- `수치·팩트`는 원문에서 기간과 출처가 확인될 때만 씁니다. 없으면 억지로 만들지 않습니다.
- `왜 지금 이 업데이트인가`에는 탐색 비용, 비교 기준, 전환 흐름, 재방문, 리뷰 신뢰, 결제 조건, 운영자 업무 중 무엇이 바뀌는지 넣습니다.
- `사용자는 무엇을 덜 해도 될까`에는 탐색, 비교, 구매, 재방문, 로그인, 알림 중 줄어드는 행동을 씁니다.
- `설계 관점`은 화면 위계, 정보구조, CTA, 상태값, 피드백, 추천 근거, 재방문 동선 중 실제로 점검할 기준을 씁니다.
- `점검 질문`은 우리 서비스에 바로 대입할 수 있는 질문으로 씁니다.

## DESIGN 출력 형식

DESIGN 글은 기능 출시처럼 억지로 `변경 전/후`를 맞추지 않습니다. 실제 화면, 표현 방식, 레이아웃, 브랜드 톤, 제작 흐름을 중심으로 씁니다.

```markdown
##### 요약

- 원문에서 중요하게 다루는 사실, 사례, 방법, 맥락
- 불릿 개수는 원문 밀도에 맞게 조정

##### 디자인 인사이트

> 저장할 이유를 압축한 한 문장

###### 왜 참고할 만한가

###### 어디에 적용할 수 있을까

###### 디자인 관점

###### 점검 질문
```

DESIGN 작성 기준:

- 본문 `요약`은 원문 내용을 압축합니다. 원문에 없는 적용처나 해석을 넣지 않습니다.
- `왜 참고할 만한가`에는 웹디자인, 브랜드 디자인, 콘텐츠 디자인, 디자인 AI 흐름 중 어디와 연결되는지 씁니다.
- `어디에 적용할 수 있을까`에는 랜딩, 상세페이지, 브랜드 캠페인, 룩북, 에디토리얼, 디자인 시스템 문서, AI 제작 흐름 등 실제 적용 범위를 씁니다.
- `디자인 관점`에는 색상, 타이포그래피, 이미지·영상 연출, 모션, 컴포넌트, 정보 위계, 브랜드 일관성 중 검수할 기준을 씁니다.
- `점검 질문`은 다음 시안이나 운영 페이지에 바로 대입할 수 있는 질문으로 씁니다.

## DEV 출력 형식

DEV 글은 구현, 접근성, 브라우저 QA, 컴포넌트, 배포 검증에서 무엇을 확인해야 하는지 보여줘야 합니다.

```markdown
##### 용어 설명

- Showit: 디자이너와 소규모 비즈니스가 웹사이트를 시각적으로 제작할 수 있게 해주는 웹사이트 빌더입니다.

##### 요약

- 원문에서 확인한 구현/검증/운영 포인트
- 기능명, API/속성, 브라우저 조건, 제한점, 검증 방법 중 하나 이상을 포함

##### 매거진 인사이트

###### 왜 지금 이 업데이트인가

###### 구현 관점

###### 실무에 어떻게 적용할 수 있을까
```

DEV 작성 기준:

- `용어 설명`은 일반 독자가 바로 알기 어려운 제품명·도구명·프로그램명만 1-3개 풀이합니다.
- `요약` 불릿은 원문에서 확인한 구체 사실을 포함해야 합니다. “~를 소개합니다”, “유용합니다” 같은 빈 문장은 실패입니다.
- 릴리즈/문서라면 기능명, 도구명, API/속성, 브라우저·접근성·성능 조건, 수치, 제한점, 검증 방법 중 하나 이상을 넣습니다.
- 블로그/오피니언 글이라면 작성자의 핵심 주장, 근거 사례, 비교 대상, 실무 판단 기준을 분리합니다.
- `구현 관점`에는 어떤 컴포넌트, 브라우저, 접근성 QA, 디자인 시스템, 배포 검증에 영향이 있는지 씁니다.
- code 스타일은 실제 코드성 표현에만 씁니다. 기능명, 제품명, 한국어 UI 문구, 강조 문장에는 쓰지 않습니다.

## 카테고리별 보강 기준

상세페이지는 같은 형식 안에서도 카테고리별로 읽을거리가 달라야 합니다. 아래 기준 중 해당 이슈에 가장 맞는 1-2개만 반영합니다.

### Service / fashion

- 브랜드관, 룩북, 카테고리 홈, 오프라인 매장, 찜/재입고처럼 `발견 → 저장 → 재방문` 흐름이 어떻게 이어지는지 봅니다.
- 할인보다 큐레이션 구조, 스타일 문맥, 브랜드 스토리, 저장 이후 알림/재방문 장치를 우선 설명합니다.
- 오프라인 이슈는 매장 규모보다 온라인 행동이 매장 이후 어떻게 회수되는지를 봅니다.

### Service / platform

- 로컬, 커뮤니티, 콘텐츠, 리서치, 검색/추천처럼 여러 참여자를 연결하는 구조를 봅니다.
- 리뷰, 후기, 프로필, 추천, 알림, 온보딩, 신고/검수 같은 신뢰 장치가 화면에서 어떻게 설명되는지 봅니다.

### Service / fintech

- 결제, 송금, 정산, 포인트, 캐시, 회비, 금융 조건처럼 돈의 흐름을 사용자가 오해하지 않게 만드는 UX를 봅니다.
- 혜택 총액보다 적용 조건, 보상 시점, 실패/취소/환불 상태, 수수료나 회비 회수 판단을 설명합니다.

### Service / ecommerce

- 멤버십, 무료배송, 페이백, 쿠폰, 리뷰, AI 검색처럼 `구매 전 불안`과 `결제 직전 계산 부담`을 낮추는 장치를 봅니다.
- AI 추천/검색은 답변 생성보다 비교 기준, 추천 근거, 상품 카드 연결, 필터 재조정 흐름을 우선합니다.

### Service / department_store

- VIP, 선물, 픽업, 예약, 오프라인 재고, 행사 동선처럼 온라인과 매장 경험이 어떻게 분담되는지 봅니다.
- 프리미엄 경험은 예약 확정, 혜택 적용, 매장 방문 전후 알림 같은 상태값이 명확한지 설명합니다.

### Service / beauty

- 피부 진단, 성분 비교, 리뷰, 랭킹, 듀프, 개인화 추천처럼 `신뢰 근거`가 화면에서 어떻게 설명되는지 봅니다.
- 글로벌/오프라인 확장은 상품 수출보다 현지 사용자가 처음 신뢰를 얻는 접점을 봅니다.

### Service / book_content

- 이어보기, 책갈피, 로그인 복구, 구독/소장/대여 결제, 추천 홈처럼 `중단 후 복귀`와 `콘텐츠 소유감`을 봅니다.

### Design

- 도구 소개보다 제작 순서, 승인 흐름, 브랜드 가이드 적용, 디자인 시스템과의 충돌, 최종 편집 가능성을 봅니다.
- AI 디자인 이슈는 결과물이 예쁜지보다 누가 검수하고, 어떤 원본/권리/컴포넌트 규칙을 기준으로 통과시키는지 설명합니다.

### DEV

- 릴리즈 소개보다 적용 조건, 영향을 받는 화면, 브라우저/런타임 호환성, 테스트와 롤백 기준을 봅니다.
- 브라우저 업데이트는 새 API 나열보다 다음 분기 컴포넌트 QA 체크리스트로 바꿉니다.

## 유익한 정보 구성

각 상세페이지는 아래 4가지 정보가 구분되어야 합니다.

- 확인된 사실: 원문에서 확인한 날짜, 기능, 범위, 수치, 제한
- 해석: 이 변화가 왜 지금 나왔고 어떤 흐름과 연결되는지
- 적용 기준: 우리 서비스에서 어떤 화면, 정책, 컴포넌트, 운영 규칙을 점검해야 하는지
- 주의점: 원문에서 확인되지 않은 수치, 과장하면 안 되는 효과, 도입 전 필요한 조건

본문에 별도 표를 만들 필요는 없지만, 문단 안에서 이 네 가지가 섞여 흐려지지 않게 씁니다.

## 설계 관점 작성 지침

`설계 관점`은 매거진 인사이트의 핵심 섹션입니다. 이 영역은 “이 기능이 생겼다”가 아니라 “이 변화에서 어떤 설계 판단을 읽어야 하는가”를 보여줘야 합니다.

반드시 포함할 관점:

- 화면 위계: 사용자가 가장 먼저 봐야 할 정보, CTA, 보조 정보의 우선순위
- 정보구조: 홈, 상세, 목록, 검색, 찜, 멤버십, 로그인 이후 화면이 어떤 순서로 이어지는지
- 행동 흐름: 탐색, 비교, 저장, 알림, 결제, 재방문 중 어느 행동을 줄이거나 강화하는지
- 판단 근거: 추천, 혜택, AI, 멤버십, 콘텐츠 카드가 사용자를 설득하는 근거를 충분히 보여주는지
- 상태와 피드백: 적용됨, 저장됨, 보상 받음, 혜택 가능, 이어보기 가능 같은 상태가 화면에서 명확한지

쓰지 말아야 할 문장:

- “사용자 경험을 개선했다”처럼 결과만 말하는 문장
- “중요하다”, “좋은 사례다”처럼 설계 근거가 없는 평가
- 기사 본문, 보도자료, 앱 릴리즈 노트를 다시 요약하는 문장
- 매출, 성장, 성과를 UX 효과처럼 단정하는 문장

## JSON 섹션 스키마 (magazine.json 출력)

마크다운 `magazine-report.md`는 `scripts/notion/export_magazine_json.py`(레거시) 또는 `scripts/collect/store.mjs`(신규)에서 아래 JSON 섹션 구조로 변환되어 `public/data/magazine.json`에 저장됩니다. AI 편집 단계에서 직접 JSON을 출력할 때도 같은 스키마를 따릅니다.

Design은 `title`을 `"디자인 인사이트"`로 두고 소제목 셋을 `왜 참고할 만한가 / 어디에 적용할 수 있을까 / 디자인 관점 / 점검 질문`으로 바꿉니다. DEV는 `매거진 인사이트` 소제목을 `왜 지금 이 업데이트인가 / 구현 관점 / 실무에 어떻게 적용할 수 있을까`로 두고, 필요하면 마지막에 `같이 보면 좋은 기술` subhead + list 블록을 둡니다.

### DEV 전용 — 용어 설명 섹션 (옵션)

원문에 일반 독자가 바로 이해하기 어려운 도구·제품명·약어가 1-3개 있을 때만 추가합니다.

```json
{
  "title": "용어 설명",
  "className": "article-section is-term-explanation",
  "prose": false,
  "blocks": [
    { "kind": "list", "html": "Radix UI: React에서 접근성 중심의 headless UI primitive를 제공하는 컴포넌트 라이브러리입니다." }
  ]
}
```

### 클래스명 규약

| className | Vue 렌더 | 용도 |
|-----------|----------|------|
| `article-section is-bullet-summary` | `<ul class="bullet-summary-list">` | `요약` 섹션 (불릿 리스트, 자체 완결 요약 원칙 적용) |
| `article-section is-deep-dive` (`prose: true`) | `<div class="section-prose">` + 블록별 렌더 | 인사이트 섹션 (quote/subhead/paragraph/list 혼합) |
| `article-section is-term-explanation` | `<ul class="term-explanation-list">` | DEV 용어 설명 |
| `article-section is-cluster-member` | 클러스터 멤버 카드 | 같은 주제 묶음 카드 (자동 생성) |
| `article-section is-related-service` | 관련 서비스 정보 | `related-service.mjs`가 자동 추가 |

### 블록 종류 (`block.kind`)

- `list` — 불릿 리스트 항목. `is-bullet-summary` 안에서는 자체 완결 요약 불릿, `is-deep-dive` 안에서는 점검 질문 항목.
- `quote` — 인사이트 리드 문장. 영역별 인사이트 섹션의 첫 블록으로만 사용합니다.
- `subhead` — 인사이트 안의 H6 소제목. 위의 잠금 소제목 셋만 허용합니다.
- `paragraph` — 일반 문단.

이 4종 외 다른 `kind`는 추가하지 않습니다.
