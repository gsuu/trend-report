# 매거진 상세 글쓰기 가이드

웹사이트 상세페이지는 뉴스 요약이 아니라, 클라이언트 미팅 전에 읽는 짧은 매거진 아티클처럼 작성합니다.

레퍼런스 톤은 대학내일 UNIV20 매거진처럼 문제의식이 분명하고, 문단 호흡이 짧고, 중간 소제목이 다음 장면으로 자연스럽게 넘어가게 만드는 방식입니다. 다만 문체를 그대로 모방하지 않고, CTTD 관점의 UIUX·서비스 해설로 바꿔 씁니다.

## 기본 구조

Service/Design은 `서비스 변화 요약`/`매거진 인사이트` 틀을 쓰고, DEV는 `기술 변화 요약`/`매거진 인사이트` 틀을 씁니다. Service/Design은 사용자 여정과 서비스 점검 질문을 중심으로 쓰고, DEV는 기술 판단과 실무 적용을 중심으로 씁니다.

목록과 상세의 제목은 발표 사실보다 독자가 확인해야 할 판단 포인트를 우선합니다. Service는 온라인 탐색, 구매, 멤버십, CRM, 전환 흐름에서 무엇을 점검해야 하는지가 보여야 합니다. Design은 화면, 브랜드, 콘텐츠 표현, 디자인 시스템, AI 제작 흐름에서 무엇을 봐야 하는지가 보여야 합니다. DEV는 구현, 접근성, 브라우저 QA, 컴포넌트, 배포 검증에서 무엇을 확인해야 하는지가 보여야 합니다. 발표 사실, 날짜, 기능명은 설명과 본문 첫 줄에 둡니다. 제목과 설명을 서로 교체하지 않고, 제목 문장만 타깃 관점으로 정리합니다.

## UIUX 아티클 구조

1. 서비스 변화 요약
   업데이트, 서비스 맥락, 변경 전/후, 수치·팩트를 한 섹션 안에 묶습니다. 수치가 없으면 억지로 쓰지 않습니다.

2. 매거진 인사이트
   bullet 요약이 아니라 짧은 매거진 아티클처럼 씁니다. 문제의식, UIUX 해석, 클라이언트 질문이 한 흐름으로 이어져야 합니다.

3. 인용형 리드
   업데이트의 의미를 한 문장으로 압축합니다. “왜 지금 봐야 하지?”가 바로 보여야 합니다.

4. 왜 지금 이 업데이트인가
   서비스가 이 기능을 낸 배경, 시장 흐름, 카테고리 맥락을 설명합니다.

5. 사용자는 무엇을 덜 해도 될까
   탐색, 비교, 구매, 재방문, 로그인, 알림 중 어느 행동이 줄거나 쉬워지는지 씁니다.

6. 설계 관점
   실제 화면과 플로우를 리뷰하듯 씁니다. 기능 설명이나 기사 요약을 반복하지 않고, 화면 위계, 정보구조, CTA, 상태값, 피드백, 추천 근거, 재방문 동선 중 무엇을 어떻게 설계해야 하는지 판단합니다.

7. 점검 질문
   서비스에 바로 대입해 볼 수 있는 질문으로 바꿉니다.

8. 마무리 관점
   “그래서 이 변화는 어떤 UX 방향을 보여주는가?”를 한두 문단으로 닫습니다.

## DEV 아티클 구조

DEV 아티클은 기술 뉴스 요약이 아니라, 프로젝트 적용 여부를 판단할 수 있는 글이어야 합니다. 아래 소제목을 고정으로 사용합니다.

1. 기술 변화 요약
   업데이트, 기술 맥락, 변경 전/후, 확인할 영향을 한 섹션 안에 묶습니다.

2. 왜 지금 이 업데이트인가
   릴리즈 배경, 생태계 변화, 기술 부채나 운영상의 문제를 설명합니다.

3. 구현 관점
   프레임워크, 브라우저, 빌드/품질 도구, 런타임, 배포 환경 중 무엇을 구현 관점에서 봐야 하는지 판단합니다.

4. 실무에 어떻게 적용할 수 있을까
   화면 구현 실무 기준으로 마크업/CSS, 접근성, 반응형, 브라우저 QA, 배포 검증 포인트를 씁니다. 반드시 `- ...` 리스트 3개 이상이어야 합니다.

5. 같이 보면 좋은 기술
   함께 검토할 API, 라이브러리, 설정, 런타임, 도구를 리스트로 씁니다. 각 항목은 `- 기술명: 왜 같이 봐야 하는지` 형식이어야 합니다.

DEV 아티클에서는 과거의 직무 지칭형 소제목이나 고객 지칭형 소제목을 쓰지 않습니다. 이 구 소제목은 빌드/뉴스레터 스크립트에서 중립 소제목으로 보정합니다.

## 문체 규칙

- 첫 문단은 기사 요약으로 시작하지 않습니다. 변화의 의미나 사용자의 장면으로 시작합니다.
- 문단은 2~4문장으로 짧게 끊습니다.
- 소제목은 설명형보다 관점형으로 씁니다. 예: `사용자는 무엇을 덜 해도 될까`
- 수치는 근거로만 씁니다. 숫자 나열로 문단을 채우지 않습니다.
- UIUX의 `핵심 업데이트`, `변경 전/후`, `수치·팩트`는 따로 쪼개지 않고 `서비스 변화 요약` 안에 묶습니다.
- DEV의 `업데이트`, `기술 맥락`, `변경 전/후`, `확인할 영향`은 `기술 변화 요약` 안에 묶습니다.
- 인사이트는 별도 bullet 섹션으로 분리하지 않고 `매거진 인사이트` 본문 안에 녹입니다.
- 매거진 인사이트는 “좋다/중요하다”로 끝내지 말고, 서비스가 점검할 질문으로 닫습니다.
- 확정되지 않은 효과는 단정하지 않습니다. 공식 수치가 없으면 성과처럼 쓰지 않습니다.
- 본문에서는 직무명이나 대상을 직접 부르지 않습니다. 메타데이터의 직무 태그는 유지하되, 본문은 `설계 관점`, `구현 관점`, `점검 질문`, `서비스에서는`처럼 화면과 판단 기준 중심으로 씁니다.
- 모든 글에 같은 정보 구성을 반복하지 않습니다. 카테고리마다 가장 유익한 실무 관점을 하나 고르고, 그 관점을 중심으로 설명합니다.
- 글을 길게 늘리기보다 `원문에서 확인된 사실`, `우리 서비스에 대입할 판단 기준`, `다음 회의에서 확인할 질문`이 분명하게 보이게 씁니다.

## 카테고리별 상세페이지 보강 기준

상세페이지는 같은 형식 안에서도 카테고리별로 읽을거리가 달라야 합니다. 아래 기준 중 해당 이슈에 가장 맞는 1~2개만 골라 반영합니다.

### Service / fashion

- 브랜드관, 룩북, 카테고리 홈, 오프라인 매장, 찜/재입고처럼 `발견 → 저장 → 재방문` 흐름이 어떻게 이어지는지 봅니다.
- 할인보다 큐레이션 구조, 스타일 문맥, 브랜드 스토리, 저장 이후 알림/재방문 장치를 우선 설명합니다.
- 오프라인 이슈는 매장 규모보다 온라인 행동이 매장 이후 어떻게 회수되는지를 봅니다.

### Service / ecommerce

- 멤버십, 무료배송, 페이백, 쿠폰, 리뷰, AI 검색처럼 `구매 전 불안`과 `결제 직전 계산 부담`을 낮추는 장치를 봅니다.
- 혜택 총량보다 사용자가 조건을 이해하는 시간, 예외 조건, 손익 판단, 장바구니/결제 화면의 반복 노출을 설명합니다.
- AI 추천/검색은 답변 생성보다 비교 기준, 추천 근거, 상품 카드 연결, 필터 재조정 흐름을 우선합니다.

### Service / department_store

- VIP, 선물, 픽업, 예약, 오프라인 재고, 행사 동선처럼 온라인과 매장 경험이 어떻게 분담되는지 봅니다.
- 프리미엄 경험은 고급스러운 표현보다 예약 확정, 혜택 적용, 매장 방문 전후 알림 같은 상태값이 명확한지 설명합니다.

### Service / beauty

- 피부 진단, 성분 비교, 리뷰, 랭킹, 듀프, 개인화 추천처럼 `신뢰 근거`가 화면에서 어떻게 설명되는지 봅니다.
- AI 진단은 진단 정확도를 단정하지 말고, 결과 카드가 제품 추천, 루틴, 재측정, 비교 가능한 수치로 이어지는지 설명합니다.
- 글로벌/오프라인 확장은 상품 수출보다 현지 사용자가 처음 신뢰를 얻는 접점을 봅니다.

### Service / book_content

- 이어보기, 책갈피, 로그인 복구, 구독/소장/대여 결제, 추천 홈처럼 `중단 후 복귀`와 `콘텐츠 소유감`을 봅니다.
- 콘텐츠 서비스는 첫 구매보다 다음 회차, 다음 권, 다음 기기에서 맥락을 잃지 않는지를 설명합니다.

### Design

- 도구 소개보다 제작 순서, 승인 흐름, 브랜드 가이드 적용, 디자인 시스템과의 충돌, 최종 편집 가능성을 봅니다.
- AI 디자인 이슈는 결과물이 예쁜지보다 누가 검수하고, 어떤 원본/권리/컴포넌트 규칙을 기준으로 통과시키는지 설명합니다.
- 디자인 회고나 매거진 글은 실제 제품 화면, 운영 규칙, 콘텐츠 톤, 디자인 조직의 의사결정으로 번역합니다.

### DEV

- 릴리즈 소개보다 적용 조건, 영향을 받는 화면, 브라우저/런타임 호환성, 테스트와 롤백 기준을 봅니다.
- AI 개발 도구는 생성 성능보다 코드 수정, 브라우저 확인, 시각 회귀, 접근성 QA, 로그/trace 공유 흐름을 설명합니다.
- 브라우저 업데이트는 새 API 나열보다 다음 분기 컴포넌트 QA 체크리스트로 바꿉니다.

## 유익한 정보 구성

각 상세페이지는 아래 4가지 정보가 구분되어야 합니다.

- 확인된 사실: 원문에서 확인한 날짜, 기능, 범위, 수치, 제한
- 해석: 이 변화가 왜 지금 나왔고 어떤 흐름과 연결되는지
- 적용 기준: 우리 서비스에서 어떤 화면, 정책, 컴포넌트, 운영 규칙을 점검해야 하는지
- 주의점: 원문에서 확인되지 않은 수치, 과장하면 안 되는 효과, 도입 전 필요한 조건

본문에 별도 표를 만들 필요는 없지만, 문단 안에서 이 네 가지가 섞여 흐려지지 않게 씁니다.

## 설계 관점 작성 지침

`설계 관점`은 매거진 인사이트의 핵심 섹션입니다. 이 영역은 “이 기능이 생겼다”가 아니라 “이 변화에서 어떤 설계 판단을 읽어야 하는가”를 보여줘야 합니다.

반드시 포함할 관점:

- 화면 위계: 사용자가 가장 먼저 봐야 할 정보, CTA, 보조 정보의 우선순위
- 정보구조: 홈, 상세, 목록, 검색, 찜, 멤버십, 로그인 이후 화면이 어떤 순서로 이어지는지
- 행동 흐름: 탐색, 비교, 저장, 알림, 결제, 재방문 중 어느 행동을 줄이거나 강화하는지
- 판단 근거: 추천, 혜택, AI, 멤버십, 콘텐츠 카드가 사용자를 설득하는 근거를 충분히 보여주는지
- 상태와 피드백: 적용됨, 저장됨, 보상 받음, 혜택 가능, 이어보기 가능 같은 상태가 화면에서 명확한지

쓰지 말아야 할 문장:

- “사용자 경험을 개선했다”처럼 결과만 말하는 문장
- “중요하다”, “좋은 사례다”처럼 설계 근거가 없는 평가
- 기사 본문, 보도자료, 앱 릴리즈 노트를 다시 요약하는 문장
- 매출, 성장, 성과를 UX 효과처럼 단정하는 문장

좋은 문장 기준:

- “디자이너는 무엇을 확인해야 하는가?”가 드러납니다.
- 화면 단위가 보입니다. 예: 홈 카드, 추천 모듈, 장바구니, 상세 CTA, 로그인 후 랜딩, 찜 목록
- 사용자의 다음 행동이 보입니다. 예: 저장한다, 비교한다, 다시 본다, 가입한다, 결제한다
- 실제 서비스에 바로 대입할 수 있습니다.

## Markdown 작성법

### UIUX 예시

```markdown
##### 서비스 변화 요약

- 업데이트:
- 서비스 맥락:
- 변경 전:
- 변경 후:
- 수치·팩트:

##### 매거진 인사이트

> 좋은 추천 UX는 첫 클릭이 아니라, 사용자가 다시 돌아왔을 때 맥락을 잃지 않게 만드는 일입니다.

###### 왜 지금 이 업데이트인가

이 업데이트는 단순한 기능 추가보다 사용자의 탐색 피로를 줄이려는 흐름에 가깝습니다.

###### 사용자는 무엇을 덜 해도 될까

사용자는 여러 화면을 오가며 후보를 비교하는 시간을 줄일 수 있습니다.

###### 설계 관점

추천 카드에는 결과만 두지 않고, 왜 이 항목이 추천됐는지 알 수 있는 기준을 함께 배치해야 합니다. 사용자는 추천 결과를 그대로 믿는 것이 아니라, 자기 기준으로 납득할 수 있을 때 다음 행동으로 넘어갑니다.

###### 점검 질문

- 우리 서비스는 사용자가 비교 기준을 직접 만들게 하고 있지 않은가?
- 추천 근거를 사용자가 납득할 수 있는 언어로 보여주고 있는가?
```

### DEV 예시

```markdown
##### 기술 변화 요약

- 업데이트: 접근성 primitive와 기존 디자인 시스템 컴포넌트를 합성하는 패턴이 확산되고 있습니다.
- 기술 맥락: 접근성 라이브러리가 기존 Button, IconButton, Link를 우회하면 디자인 시스템이 둘로 갈라질 수 있습니다.
- 변경 전: trigger 컴포넌트가 별도 DOM을 만들면서 button 안의 button 같은 중첩 문제가 생길 수 있었습니다.
- 변경 후: asChild 패턴과 ref forwarding을 통해 접근성 동작과 디자인 시스템 DOM을 같은 노드에 붙일 수 있습니다.
- 확인할 영향: 공통 컴포넌트의 ref 전달, role 유지, disabled 처리, 키보드 포커스 QA를 함께 점검해야 합니다.

##### 매거진 인사이트

> 접근성 라이브러리는 디자인 시스템을 대체하는 도구가 아니라, 기존 컴포넌트에 올바른 동작을 붙이는 도구여야 합니다.

###### 왜 지금 이 업데이트인가

접근성 primitive를 도입하는 팀이 늘면서 기존 디자인 시스템과 어떻게 합성할지가 중요해지고 있습니다.

###### 구현 관점

asChild는 강력하지만 잘못 쓰면 interactive element 안에 interactive element를 중첩하는 문제가 생길 수 있습니다. 팀은 button과 a의 역할, disabled 처리, ref forwarding을 컴포넌트 규칙으로 정리해야 합니다.

###### 실무에 어떻게 적용할 수 있을까

- 공통 Button과 Link가 forwardRef를 지원하는지 먼저 점검해 접근성 primitive와 디자인 시스템 컴포넌트가 같은 DOM 노드를 쓰게 합니다.
- Dialog Trigger, Tooltip Trigger, Dropdown Trigger처럼 사용자 조작 시작점부터 asChild 패턴을 적용해 중복 마크업을 줄입니다.
- QA에서는 button 안의 button, a 안의 button처럼 interactive element 중첩이 생기지 않는지 키보드 포커스와 함께 확인합니다.

###### 같이 보면 좋은 기술

- Radix UI `asChild`: 기존 디자인 시스템 컴포넌트와 접근성 primitive를 합성하는 핵심 패턴입니다.
- React `forwardRef`: trigger 컴포넌트가 실제 DOM 노드에 접근해야 하므로 ref 전달 규칙을 먼저 확인해야 합니다.
- ARIA roles: polymorphic 컴포넌트가 button과 link 역할을 바꿀 때 접근성 의미가 깨지지 않는지 검증해야 합니다.
```
