# 매거진 상세 글쓰기 가이드

웹사이트 상세페이지는 뉴스 요약이 아니라, 클라이언트 미팅 전에 읽는 짧은 매거진 아티클처럼 작성합니다.

레퍼런스 톤은 대학내일 UNIV20 매거진처럼 문제의식이 분명하고, 문단 호흡이 짧고, 중간 소제목이 다음 장면으로 자연스럽게 넘어가게 만드는 방식입니다. 다만 문체를 그대로 모방하지 않고, CTTD 관점의 UIUX·서비스 해설로 바꿔 씁니다.

## 독자 타겟

이 매거진의 1차 독자는 CTTD가 클라이언트 미팅에서 함께 이야기할 국내 웹서비스 실무자입니다. 구체적으로는 이커머스, 패션, 뷰티, 콘텐츠, 플랫폼, 핀테크 서비스를 운영하거나 개편하는 웹기획자, PM, 웹디자이너, UIUX 개발자를 기준으로 씁니다.

독자가 기대하는 것은 “새 기능이 나왔다”가 아니라 “우리 서비스의 화면, 정책, 전환 흐름, 디자인 시스템, 구현·QA 기준에서 무엇을 점검해야 하는가”입니다. 따라서 채택 후보는 아래 질문 중 하나 이상에 답해야 합니다.

- 웹기획자/PM이 서비스 구조, 혜택 조건, 검색·추천, 결제, 재방문, CRM 흐름에 바로 대입할 수 있는가?
- 웹디자이너가 화면 위계, 정보구조, 콘텐츠 표현, 브랜드 경험, 디자인 시스템, AI 제작·검수 흐름의 레퍼런스로 볼 수 있는가?
- UIUX 개발자가 CSS/HTML/JavaScript, 접근성, 브라우저 호환성, 컴포넌트 구현, 디자인-코드 연동, 테스트·QA 기준으로 바로 확인할 수 있는가?

반대로 일반 소비자용 AI 앱 기능 묶음, 모델 성능 발표, 백엔드/API 운영, 클라우드 관리자 도구, OS 설정, 단순 프로모션처럼 위 독자가 자기 서비스 회의에서 물어볼 화면·플로우·정책·구현 질문으로 바뀌지 않는 이슈는 제외합니다. 유명 브랜드나 최신 AI 소식이어도 독자 타겟의 실무 판단으로 연결되지 않으면 메인 아티클로 만들지 않습니다.

## 기본 구조

Service/Design은 `서비스 변화 요약`/`매거진 인사이트` 틀을 쓰고, DEV는 `요약`/`매거진 인사이트` 틀을 씁니다. Service/Design은 사용자 여정과 서비스 점검 질문을 중심으로 쓰고, DEV는 기술 판단과 실무 적용을 중심으로 씁니다.

목록과 상세의 제목은 발표 사실보다 독자가 확인해야 할 판단 포인트를 우선합니다. Service는 온라인 탐색, 구매, 멤버십, CRM, 전환 흐름에서 무엇을 점검해야 하는지가 보여야 합니다. Design은 화면, 브랜드, 콘텐츠 표현, 디자인 시스템, AI 제작 흐름에서 무엇을 봐야 하는지가 보여야 합니다. DEV는 구현, 접근성, 브라우저 QA, 컴포넌트, 배포 검증에서 무엇을 확인해야 하는지가 보여야 합니다. 발표 사실, 날짜, 기능명은 설명과 본문 첫 줄에 둡니다. 제목과 설명을 서로 교체하지 않고, 제목 문장만 타깃 관점으로 정리합니다.

상단 요약설명은 원문을 직접 요약하되 `어떤 브랜드에서`, `무엇을 발표했는지`, `왜 이 발표가 나왔는지`가 보이게 씁니다. 제목처럼 해석 포인트를 앞세우거나, `업데이트` 문구를 그대로 반복하지 않습니다. 좋은 설명은 `브랜드 + 발표 내용 + 배경/문제 맥락`의 2문장 안팎입니다.

요약설명을 쓴 뒤에는 반드시 한 번 더 읽고 어색한 부분을 고칩니다. 특히 `브랜드명+조사`가 어색하면 `브랜드에서`로 바꾸고, 외국어 서비스명 뒤에 억지 조사를 붙이지 않습니다. `무엇을 발표했는지`와 `왜 발표했는지`가 같은 문장으로 뭉개지면 두 문장으로 나누고, 첫 업데이트 문구와 거의 같으면 설명을 다시 씁니다.

플랫폼명과 출처명을 혼동하지 않습니다. `토스 테크`, `Google Blog`, `Figma Blog`처럼 글이 올라온 매체명은 출처로 두고, 요약설명과 아티클 브랜드에는 실제 발표 주체나 제품명을 씁니다. 예를 들어 `토스 테크는 토스플레이스가...`가 아니라 `토스플레이스가...`로 시작해야 합니다.

제목 보정은 코드의 개별 영어 제목 치환 규칙으로 처리하지 않습니다. AI/편집 단계에서 원문을 확인한 뒤 이 가이드에 맞춰 작성합니다. 자동화 코드는 원문 제목 정리, 뉴스레터 이슈명 보존, 빈 제목 방지까지만 담당합니다.

본문에서 code 스타일은 실제 코드성 표현에만 씁니다. 명령어, 파일명, 패키지명, API/속성명, CSS/HTML/ARIA 속성, 설정값처럼 개발자가 그대로 입력하거나 확인할 수 있는 표현은 code로 둘 수 있습니다. 기능명, 제품명, 서비스명, 한국어 문구, 인용문, 강조 표현에는 백틱이나 `<code>`를 쓰지 않습니다.

## 카테고리별 수집 기준

후보 수집과 채택 기준은 [데이터 수집 전략](data-collection-strategy.md)을 따릅니다. 이 문서는 이미 채택된 후보를 매거진 글로 바꾸는 작성 기준만 다룹니다.

DEV 글은 [DEV Digest Agent Prompt](dev-digest-agent-prompt.md)의 출력 형식을 우선합니다. 이 가이드는 문체, 독자 관점, 중복 문장 방지 기준만 보완합니다.

## Service/Design 아티클 구조

1. 서비스 변화 요약
   `업데이트`, `서비스 맥락`, `변경 전`, `변경 후`, `수치·팩트` 라벨을 유지합니다. 특히 `업데이트`는 한 줄짜리 제목 반복이 아니라 원문을 읽은 느낌이 나도록 2줄 정도로 씁니다. 수치가 없으면 억지로 쓰지 않습니다.

2. 매거진 인사이트
   bullet 요약이 아니라 짧은 매거진 아티클처럼 씁니다. 문제의식, UIUX 해석, 클라이언트 질문이 한 흐름으로 이어져야 합니다. Service는 사용자 여정과 정책 판단을, Design은 화면·브랜드·제작 흐름과 검수 기준을 중심으로 씁니다.

3. 인용형 리드
   업데이트의 의미를 한 문장으로 압축합니다. “왜 지금 봐야 하지?”가 바로 보여야 합니다.

4. 왜 지금 이 업데이트인가
   서비스가 이 기능을 낸 배경, 시장 흐름, 카테고리 맥락을 3줄 정도로 설명합니다. Service 글은 반드시 웹서비스 전문가가 주목해야 할 이유가 보여야 합니다. 예를 들어 탐색 비용, 비교 기준, 전환 흐름, 재방문, 리뷰 신뢰, 결제 조건, 운영자 업무 중 무엇이 바뀌는지 명확히 씁니다. Design 글은 제작 도구, 브랜드 시스템, 콘텐츠 표현, 디자인 시스템 운영 흐름 중 무엇과 연결되는지 함께 씁니다.

5. 사용자는 무엇을 덜 해도 될까
   탐색, 비교, 구매, 재방문, 로그인, 알림 중 어느 행동이 줄거나 쉬워지는지 3줄 정도로 씁니다. Design 글은 사용자의 이해, 신뢰 형성, 콘텐츠 판단, 제작자의 검수·승인 부담 중 무엇이 줄어드는지 설명합니다.

6. 설계 관점
   실제 화면과 플로우를 리뷰하듯 3줄 정도로 씁니다. 기능 설명이나 기사 요약을 반복하지 않고, 화면 위계, 정보구조, CTA, 상태값, 피드백, 추천 근거, 재방문 동선 중 무엇을 어떻게 설계해야 하는지 판단합니다. Service 글은 우리 서비스에 적용할 때 고려할 조건, 예외, 검수 지점이 반드시 들어가야 합니다. Design 글은 색상, 타이포그래피, 컴포넌트, 콘텐츠 톤, 이미지·영상 생성 흐름, 브랜드 일관성 중 실제 검수 기준을 드러냅니다.

7. 점검 질문
   서비스에 바로 대입해 볼 수 있는 질문으로 바꿉니다.

8. 마무리 관점
   “그래서 이 변화는 어떤 UX 방향을 보여주는가?”를 한두 문단으로 닫습니다.

## DEV 아티클 구조

DEV 아티클의 필수 출력 형식, 후보 선별, 블로그/오피니언 처리, `요약` 섹션 작성 기준은 [DEV Digest Agent Prompt](dev-digest-agent-prompt.md)를 우선합니다.

이 가이드에서는 DEV 글에도 공통 적용되는 문체만 보완합니다. 기술 뉴스 요약이 아니라 프로젝트 적용 여부를 판단할 수 있는 글이어야 하며, 구현·접근성·브라우저 QA·컴포넌트·배포 검증 중 무엇을 확인해야 하는지 보여야 합니다.

## 큐레이션·릴리즈 노트 처리

큐레이션, 뉴스레터, 팟캐스트, 릴리즈 노트 기반 DEV 글의 세부 선별 기준은 [DEV Digest Agent Prompt](dev-digest-agent-prompt.md)를 따릅니다.

발행 본문에는 편집자가 어떻게 선별했는지 쓰지 않습니다. 원문에서 확인한 핵심 항목과 그 항목이 어떤 화면, 컴포넌트, 운영 기준과 연결되는지만 씁니다.

## 제외 기준

아래 항목은 원문이 있더라도 기본적으로 발행본에서 제외합니다.

- 독립 블로그/매거진 글이 특정 도구, 서비스, 강의, 컨설팅을 홍보하는 성격이 강한 경우
- 제목은 좋아 보이지만 실제 내용이 방법론 일반론이거나 광고성 사례에 가까운 경우
- 백엔드/API/보안 운영이 중심이고 화면, 컴포넌트, 접근성 QA, 서비스 플로우와 직접 연결하기 어려운 경우
- Service 후보인데 웹서비스 전문가가 주목해야 할 이유와 적용 시 고려사항을 각각 한 문장 이상으로 쓸 수 없는 경우
- 자동 생성 문장으로만 설명 가능하고 원문에서 확인한 구체 항목이 3개 이상 나오지 않는 경우
- 카드 제휴, 쿠폰/e쿠폰 거래, 멤버십 프로모션, 콘텐츠 파트너십, 외부 AI 추천 연동처럼 보도자료 문구만으로는 화면 변화가 보이지 않는 경우
- 글의 핵심이 아니라 작성자가 덧붙인 일반 UX 교훈으로만 `매거진 인사이트`를 만들 수 있는 경우

약한 후보를 살리려고 문장을 늘리지 않습니다. `혜택 조건을 헷갈리지 않게`, `결제 후 다음 행동으로 연결`, `추천을 안전하게 데려오기`처럼 어느 서비스에도 붙일 수 있는 문장만 나오면 본문을 쓰지 말고 제외 메모로 보냅니다.

## 작성 후 리뷰 에이전트

20~30개 매거진 후보 문구를 작성한 뒤에는 반드시 별도 리뷰 에이전트가 원문을 다시 열어 대조합니다. 리뷰 에이전트는 문장 톤보다 사실 일치와 원문 반영도를 먼저 봅니다.

- 각 글의 `출처 URL`을 열어 원문 제목, 날짜, 기능명, 도구명, API명, 수치가 본문과 맞는지 확인합니다.
- `서비스 변화 요약` 또는 DEV `요약`에 원문에서 확인한 구체 항목이 3개 이상 들어갔는지 확인합니다.
- 제목과 deck이 원문의 핵심을 담는지 확인합니다. 원문이 AccessLint MCP 글인데 일반적인 “접근성 자동화” 글처럼 흐려지면 수정 필요입니다.
- 카테고리가 잘못된 글은 발행 전에 고칩니다. 예: 디자인 AI는 `design/ai`, 개발 AI는 `dev/ai`, 화면 구현과 무관한 DATA/API 중심 글은 제외합니다.
- 광고, 스폰서, 홍보성 튜토리얼, 백엔드/API 중심 글은 매거진 본문에서 제외하고 `수집했지만 제외한 것`에 남깁니다.
- 원문에서 확인된 화면·플로우 없이 제휴, 프로모션, 안전거래, 외부 AI 연동을 UX 일반론으로 해석한 글은 제외합니다.
- Service 글은 `왜 지금 이 업데이트인가`에서 웹서비스 실무자가 봐야 할 이유가 드러나는지, `설계 관점`에서 우리 서비스에 적용할 때 고려할 화면·정책·플로우 조건이 드러나는지 확인합니다. 둘 중 하나라도 없으면 제외 또는 수정 필요입니다.
- 리뷰 결과가 `수정 필요`이면 리뷰 에이전트가 원문 기준 수정안을 만들고, 제목·deck·요약·본문·카테고리·태그를 다시 고친 뒤 재검수합니다.
- 리뷰 결과가 `제외`이면 해당 후보를 `수집했지만 제외한 것`으로 옮기고 제외 사유와 다시 볼 조건을 남깁니다.

리뷰 역할과 판정 형식은 [매거진 에이전트 운영 흐름](magazine-agent-workflow.md)을 따릅니다.

## 문체 규칙

- 첫 문단은 기사 요약으로 시작하지 않습니다. 변화의 의미나 사용자의 장면으로 시작합니다.
- 문단은 2~4문장으로 짧게 끊습니다.
- 요약 섹션의 라벨은 유지합니다. 다만 `업데이트`는 제목 복사가 아니라 원문 핵심을 2줄 정도로 풀어 씁니다.
- Service/Design의 `왜 지금 이 업데이트인가`, `사용자는 무엇을 덜 해도 될까`, `설계 관점`은 각각 3줄 정도로 씁니다. 한 문장 감상으로 끝내지 말고, 원문에 나온 기능명·출시 범위·화면 변화·정책 변화·제작 조건을 포함합니다.
- DEV의 `왜 지금 이 업데이트인가`, `구현 관점`은 각각 3줄 정도로 씁니다. 한 문장 감상으로 끝내지 말고, 원문에 나온 기술명·버전·기능·적용 조건을 포함합니다.
- 소제목은 설명형보다 관점형으로 씁니다. 예: `사용자는 무엇을 덜 해도 될까`
- 수치는 근거로만 씁니다. 숫자 나열로 문단을 채우지 않습니다.
- UIUX의 `핵심 업데이트`, `변경 전/후`, `수치·팩트`는 따로 쪼개지 않고 `서비스 변화 요약` 안에 묶습니다.
- DEV의 `업데이트`, `기술 맥락`, `변경 전/후`, `확인할 영향`은 `요약` 안에 묶습니다. 블로그 글에서는 `작성자 관점`, `근거·사례`, `참고할 점`을 한 카드 안에 묶습니다.
- 인사이트는 별도 bullet 섹션으로 분리하지 않고 `매거진 인사이트` 본문 안에 녹입니다.
- 매거진 인사이트는 “좋다/중요하다”로 끝내지 말고, 서비스가 점검할 질문으로 닫습니다.
- 확정되지 않은 효과는 단정하지 않습니다. 공식 수치가 없으면 성과처럼 쓰지 않습니다.
- 본문에서는 직무명이나 대상을 직접 부르지 않습니다. 메타데이터의 직무 태그는 유지하되, 본문은 `설계 관점`, `구현 관점`, `점검 질문`, `서비스에서는`처럼 화면과 판단 기준 중심으로 씁니다.
- 각 아티클의 `태그`는 글에서 가장 중요한 단어 3개만 씁니다. 직무 태그와 중복시키지 말고, 기능명·제품명·설계 쟁점·기술 쟁점 중심의 짧은 명사로 고릅니다.
- 모든 글에 같은 정보 구성을 반복하지 않습니다. 카테고리마다 가장 유익한 실무 관점을 하나 고르고, 그 관점을 중심으로 설명합니다.
- 글을 길게 늘리기보다 `원문에서 확인된 사실 요약`, `우리 서비스에 대입할 판단 기준`, `다음 회의에서 확인할 질문`이 분명하게 보이게 씁니다.

## 해외 원문 한국어 패치

외국어 원문은 직역하지 않고, 원문의 팩트를 유지한 채 한국어 독자가 자연스럽게 읽는 문장으로 다시 씁니다. 번역문처럼 보이는 표현보다 국내 서비스 기획·디자인·개발 실무에서 쓰는 말로 패치합니다.

- 원문 문장 구조를 따라가지 않습니다. 영어의 긴 관계절, 수동태, 명사 나열은 한국어에서 `누가/무엇이/어떤 화면에서/무엇을 바꾸는지` 순서로 다시 씁니다.
- 제품명, 기능명, API명, 공식 캠페인명은 원문 표기를 유지합니다. 단, 처음 한 번은 한국어 설명을 붙입니다. 예: `Agent Inbox`는 `에이전트 결과를 사람이 검토하는 받은편지함 형태의 기능`처럼 풉니다.
- 에피소드 번호, 이슈 번호, 릴리즈 번호는 원문의 표기 체계를 보존합니다. 번호 자체를 `회`, `호`로 직역하면 한국어 독자에게 방송 회차나 잡지 호수처럼 어색하게 읽힐 수 있으므로, 브랜드명과 함께 `#999`, `Issue 782`, `TP 242`처럼 둡니다.
- `seamless`, `empower`, `unlock`, `reimagine`, `delight`, `next-generation` 같은 마케팅 표현은 그대로 옮기지 않습니다. 실제 화면 변화나 작업 흐름 변화로 바꿉니다.
- `users can...`은 `사용자는 ...할 수 있습니다`만 반복하지 말고, `사용자가 직접 비교하던 과정을 줄입니다`, `선택 기준을 화면 안에서 확인할 수 있습니다`처럼 행동 변화로 씁니다.
- `원문은 ... 다룹니다`, `... 흐름을 보여줍니다`, `... 방식으로 소개됩니다` 같은 요약투를 반복하지 않습니다. `어떤 브랜드가 / 어떤 기능·글을 냈고 / 왜 지금 봐야 하는지`를 한 번 쓴 뒤, 두 번째 문장에서 실무자가 볼 판단 기준으로 이어갑니다.
- 영어 용어를 그대로 나열하지 않습니다. `localization support`는 `언어별 표시 정보 지원`, `lazy loading`은 첫 등장 뒤 `지연 로딩`, `show notes`는 `쇼노트`, `global/local solution`은 `전역/지역 해법`처럼 한국어 독자가 바로 이해할 표현으로 바꿉니다.
- 단, API·속성·명령어·라이브러리 이름은 원문 표기를 유지합니다. `name_localized`, `fetchpriority`, `CSS Modules`, `Bun 1.3.13`처럼 검색해야 하는 기술명은 번역하지 않고, 주변 문장만 자연스럽게 풉니다.
- 외국어 원문의 과장된 효과는 낮춰 씁니다. 원문이 성과 수치를 제시하지 않으면 `전환이 오른다`, `생산성이 개선된다`처럼 단정하지 않습니다.
- 한국어 문장은 한 문장에 한 판단만 담습니다. 쉼표로 길게 이어지는 번역체 문장은 두 문장으로 나눕니다.
- UIUX 용어는 국내 실무 표현을 우선합니다. `journey`는 문맥에 따라 `여정`, `흐름`, `구매 전후 과정`으로, `engagement`는 `참여`보다 `재방문`, `반응`, `사용 빈도`로 바꿉니다.
- DEV 원문은 API 이름과 기술 용어를 보존하되, 의미는 `어떤 컴포넌트/브라우저/QA에 영향이 있는지`로 풀어 씁니다.
- 원문에 없는 국내 사례나 수치를 보태지 않습니다. 한국어로 자연스럽게 바꾸더라도 팩트 범위는 원문 안에 머뭅니다.

번역 전후 체크:

- 직역 문장: `Spotify enables users to discover audio through conversational prompts.`
- 좋은 한국어 패치: `Spotify는 사용자가 장르나 카테고리를 먼저 고르기보다, 듣고 싶은 상황을 말하면서 음악과 팟캐스트를 찾는 흐름을 열었습니다.`
- 피해야 할 문장: `Spotify는 대화형 프롬프트를 통해 사용자가 오디오를 발견할 수 있도록 합니다.`

## 카테고리별 상세페이지 보강 기준

상세페이지는 같은 형식 안에서도 카테고리별로 읽을거리가 달라야 합니다. 아래 기준 중 해당 이슈에 가장 맞는 1~2개만 골라 반영합니다.

### Service / fashion

- 브랜드관, 룩북, 카테고리 홈, 오프라인 매장, 찜/재입고처럼 `발견 → 저장 → 재방문` 흐름이 어떻게 이어지는지 봅니다.
- 할인보다 큐레이션 구조, 스타일 문맥, 브랜드 스토리, 저장 이후 알림/재방문 장치를 우선 설명합니다.
- 오프라인 이슈는 매장 규모보다 온라인 행동이 매장 이후 어떻게 회수되는지를 봅니다.

### Service / platform

- 로컬, 커뮤니티, 콘텐츠, 리서치, 검색/추천처럼 여러 참여자를 연결하는 구조를 봅니다.
- 중요한 기준은 기능 출시보다 참여자가 더 쉽게 입력하고, 찾고, 신뢰하고, 다시 돌아오는 흐름입니다.
- 리뷰, 후기, 프로필, 추천, 알림, 온보딩, 신고/검수 같은 신뢰 장치가 화면에서 어떻게 설명되는지 봅니다.

### Service / fintech

- 결제, 송금, 정산, 포인트, 캐시, 회비, 금융 조건처럼 돈의 흐름을 사용자가 오해하지 않게 만드는 UX를 봅니다.
- 혜택 총액보다 적용 조건, 보상 시점, 실패/취소/환불 상태, 수수료나 회비 회수 판단을 설명합니다.
- 가맹점·사장님·운영자 도구는 고객 화면이 아니어도 확인, 승인, 조치가 줄어드는 운영 UX로 봅니다.

### Service / ecommerce

- 멤버십, 무료배송, 페이백, 쿠폰, 리뷰, AI 검색처럼 `구매 전 불안`과 `결제 직전 계산 부담`을 낮추는 장치를 봅니다.
- 혜택 총량보다 사용자가 조건을 이해하는 시간, 예외 조건, 손익 판단, 장바구니/결제 화면의 반복 노출을 설명합니다.
- AI 추천/검색은 답변 생성보다 비교 기준, 추천 근거, 상품 카드 연결, 필터 재조정 흐름을 우선합니다.

### Service / department_store

- VIP, 선물, 픽업, 예약, 오프라인 재고, 행사 동선처럼 온라인과 매장 경험이 어떻게 분담되는지 봅니다.
- 프리미엄 경험은 고급스러운 표현보다 예약 확정, 혜택 적용, 매장 방문 전후 알림 같은 상태값이 명확한지 설명합니다.

### Service / beauty

- 피부 진단, 성분 비교, 리뷰, 랭킹, 듀프, 개인화 추천처럼 `신뢰 근거`가 화면에서 어떻게 설명되는지 봅니다.
- AI 진단은 진단 정확도를 단정하지 말고, 결과 카드가 제품 추천, 루틴, 재측정, 비교 가능한 수치로 이어지는지 설명합니다.
- 글로벌/오프라인 확장은 상품 수출보다 현지 사용자가 처음 신뢰를 얻는 접점을 봅니다.

### Service / book_content

- 이어보기, 책갈피, 로그인 복구, 구독/소장/대여 결제, 추천 홈처럼 `중단 후 복귀`와 `콘텐츠 소유감`을 봅니다.
- 콘텐츠 서비스는 첫 구매보다 다음 회차, 다음 권, 다음 기기에서 맥락을 잃지 않는지를 설명합니다.

### Design

- 도구 소개보다 제작 순서, 승인 흐름, 브랜드 가이드 적용, 디자인 시스템과의 충돌, 최종 편집 가능성을 봅니다.
- AI 디자인 이슈는 결과물이 예쁜지보다 누가 검수하고, 어떤 원본/권리/컴포넌트 규칙을 기준으로 통과시키는지 설명합니다.
- 디자인 회고나 매거진 글은 실제 제품 화면, 운영 규칙, 콘텐츠 톤, 디자인 조직의 의사결정으로 번역합니다.

### DEV

- 릴리즈 소개보다 적용 조건, 영향을 받는 화면, 브라우저/런타임 호환성, 테스트와 롤백 기준을 봅니다.
- AI 개발 도구는 생성 성능보다 코드 수정, 브라우저 확인, 시각 회귀, 접근성 QA, 로그/trace 공유 흐름을 설명합니다.
- 브라우저 업데이트는 새 API 나열보다 다음 분기 컴포넌트 QA 체크리스트로 바꿉니다.

## 유익한 정보 구성

각 상세페이지는 아래 4가지 정보가 구분되어야 합니다.

- 확인된 사실: 원문에서 확인한 날짜, 기능, 범위, 수치, 제한
- 해석: 이 변화가 왜 지금 나왔고 어떤 흐름과 연결되는지
- 적용 기준: 우리 서비스에서 어떤 화면, 정책, 컴포넌트, 운영 규칙을 점검해야 하는지
- 주의점: 원문에서 확인되지 않은 수치, 과장하면 안 되는 효과, 도입 전 필요한 조건

본문에 별도 표를 만들 필요는 없지만, 문단 안에서 이 네 가지가 섞여 흐려지지 않게 씁니다.

## 설계 관점 작성 지침

`설계 관점`은 매거진 인사이트의 핵심 섹션입니다. 이 영역은 “이 기능이 생겼다”가 아니라 “이 변화에서 어떤 설계 판단을 읽어야 하는가”를 보여줘야 합니다.

반드시 포함할 관점:

- 화면 위계: 사용자가 가장 먼저 봐야 할 정보, CTA, 보조 정보의 우선순위
- 정보구조: 홈, 상세, 목록, 검색, 찜, 멤버십, 로그인 이후 화면이 어떤 순서로 이어지는지
- 행동 흐름: 탐색, 비교, 저장, 알림, 결제, 재방문 중 어느 행동을 줄이거나 강화하는지
- 판단 근거: 추천, 혜택, AI, 멤버십, 콘텐츠 카드가 사용자를 설득하는 근거를 충분히 보여주는지
- 상태와 피드백: 적용됨, 저장됨, 보상 받음, 혜택 가능, 이어보기 가능 같은 상태가 화면에서 명확한지

쓰지 말아야 할 문장:

- “사용자 경험을 개선했다”처럼 결과만 말하는 문장
- “중요하다”, “좋은 사례다”처럼 설계 근거가 없는 평가
- 기사 본문, 보도자료, 앱 릴리즈 노트를 다시 요약하는 문장
- 매출, 성장, 성과를 UX 효과처럼 단정하는 문장

좋은 문장 기준:

- “디자이너는 무엇을 확인해야 하는가?”가 드러납니다.
- 화면 단위가 보입니다. 예: 홈 카드, 추천 모듈, 장바구니, 상세 CTA, 로그인 후 랜딩, 찜 목록
- 사용자의 다음 행동이 보입니다. 예: 저장한다, 비교한다, 다시 본다, 가입한다, 결제한다
- 실제 서비스에 바로 대입할 수 있습니다.

##### 매거진 인사이트

> 좋은 추천 UX는 첫 클릭이 아니라, 사용자가 다시 돌아왔을 때 맥락을 잃지 않게 만드는 일입니다.

###### 왜 지금 이 업데이트인가

이 업데이트는 단순한 기능 추가보다 사용자의 탐색 피로를 줄이려는 흐름에 가깝습니다.
원문에서 확인한 출시 범위와 화면 변화가 어떤 맥락에서 나왔는지 함께 설명합니다.
사용자가 이전보다 덜 반복하거나 더 쉽게 판단하게 되는 지점을 중심에 둡니다.

###### 사용자는 무엇을 덜 해도 될까

사용자는 여러 화면을 오가며 후보를 비교하는 시간을 줄일 수 있습니다.

###### 설계 관점

추천 카드에는 결과만 두지 않고, 왜 이 항목이 추천됐는지 알 수 있는 기준을 함께 배치해야 합니다. 사용자는 추천 결과를 그대로 믿는 것이 아니라, 자기 기준으로 납득할 수 있을 때 다음 행동으로 넘어갑니다.

###### 점검 질문

- 우리 서비스는 사용자가 비교 기준을 직접 만들게 하고 있지 않은가?
- 추천 근거를 사용자가 납득할 수 있는 언어로 보여주고 있는가?
