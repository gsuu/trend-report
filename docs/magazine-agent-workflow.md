# 매거진 에이전트 운영 흐름

매거진 발행은 한 번에 쓰지 않고 `수집 → 원문 검증 → 분류 → 글쓰기 → 리뷰/수정 → QA` 역할을 순서대로 통과시킵니다. 각 역할은 이전 단계의 산출물을 믿되, 자기 단계에서 반드시 확인해야 하는 기준을 따로 가집니다.

## 1. 수집 에이전트

목표는 후보를 넓게 모으는 것입니다. 매주 Service / Design / DEV를 합쳐 20~30개 후보를 만들고, 각 후보에 최종 기준 원문 또는 후보 발견 출처를 남깁니다.

필수 산출물:

- 후보 제목, 플랫폼, 대분류, 소분류, 직무 태그
- 후보 발견 출처와 최종 기준 원문 URL
- 원문에서 확인한 기능, 정책, 수치, 날짜, 제한 사항
- 이미지 후보와 이미지 출처
- 채택/보류/제외 초기 판단

수집 에이전트는 후보를 평가하지 말고 증거를 모읍니다. 공식 출처를 못 찾은 후보는 `원문 미확인`으로 표시하고 글쓰기 단계로 넘기지 않습니다.

## 2. 원문 검증 에이전트

목표는 후보 발견 출처와 최종 기준 원문을 분리하는 것입니다. 이 단계는 글쓰기 전에 수행합니다.

검증 기준:

- URL이 실제 최종 기준 원문인지 확인합니다. 큐레이션, 개인 블로그, YouTube 요약, 보도자료 재인용 글은 후보 발견 출처로만 둡니다.
- 원문 안에서 발행에 쓸 수 있는 구체 사실 3개 이상을 추출합니다.
- 원문이 홍보성 튜토리얼, 스폰서 콘텐츠, 백엔드/API 중심 글이면 발행 후보에서 제외합니다.
- 원문 제목, 작성자/기관, 발행일, 핵심 기능명, 수치, 제한 사항을 기록합니다.

원문 검증 에이전트가 `원문 부족`으로 판정한 후보는 글쓰기 에이전트로 넘기지 않습니다. 대신 `수집했지만 제외한 것`에 남길 제외 사유를 작성합니다.

## 3. 분류 에이전트

목표는 후보 20~30개를 Service / Design / DEV와 각 소분류에 맞게 정리하는 것입니다.

분류 기준:

- Service: 사용자 여정, 혜택, 탐색, 구매, 재방문, 운영 UX 변화
- Design: 화면, 브랜드, 콘텐츠 표현, 디자인 시스템, AI 디자인 제작 흐름
- DEV: 화면 구현, 접근성 QA, 브라우저/웹 플랫폼, 컴포넌트, 빌드/테스트, AI 개발 도구

분류 에이전트는 `dev/data_api`, `dev/ai_design`, `dev/ai_dev`처럼 중복되거나 역할이 섞인 소분류를 만들지 않습니다. 개발 AI는 `dev/ai`, 디자인 AI는 `design/ai`로 정리합니다. 데이터/API/백엔드 중심이고 화면 구현이나 접근성 QA에 직접 연결되지 않으면 발행 후보에서 제외합니다.

## 4. 글쓰기 에이전트

목표는 채택 후보를 `runs/YYYY-MM-DD/magazine-report.md`에 매거진 문장으로 작성하는 것입니다.

글쓰기 규칙:

- 반드시 최종 기준 원문을 읽고 씁니다.
- 원문에서 확인한 구체 사실 3개 이상을 `서비스 변화 요약` 또는 `기술 변화 요약`에 반영합니다.
- 제목은 발표 사실보다 독자가 확인해야 할 판단 포인트를 우선합니다.
- DEV 글은 `기술 변화 요약`, `왜 지금 이 업데이트인가`, `구현 관점`, `실무에 어떻게 적용할 수 있을까`, `같이 보면 좋은 기술` 흐름을 지킵니다.
- 큐레이션/릴리즈 노트 글은 “뉴스레터가 나왔다”가 아니라 원문 안에서 이번 주 실무자가 확인할 항목을 골라 씁니다.

글쓰기 에이전트는 원문에 없는 효과, 국내 사례, 수치를 보태지 않습니다. 원문이 홍보성·일반론에 그치면 글을 억지로 만들지 않고 제외 후보로 돌립니다.

## 5. 리뷰 에이전트

목표는 작성된 20~30개 매거진 문구를 원문과 대조하고, 문제가 있는 글을 발행 가능한 수준으로 다시 고치는 것입니다. 리뷰 에이전트는 문장 교정자가 아니라 원문 대조 검수자이자 수정 지시자입니다.

리뷰 에이전트는 각 글마다 반드시 출처 URL을 열고 아래 항목을 확인합니다.

1. 원문 일치: 제목, 플랫폼, 기능명, 날짜, 수치, 도구명, API명이 원문과 맞는가.
2. 구체성: 원문에서 확인한 구체 항목이 3개 이상 본문에 들어갔는가.
3. 카테고리: Service / Design / DEV와 소분류가 실제 내용과 맞는가.
4. 원문 반영도: 글이 원문 핵심을 다루는가, 아니면 일반론으로 바뀌었는가.
5. 실무 연결: UIUX 서비스 설계, 디자인 제작, 화면 구현/QA 중 하나로 바로 연결되는가.
6. 과장 방지: 원문에 없는 성과, 전환율, 생산성, 자동화 효과를 단정하지 않았는가.
7. 제외 판단: 광고, 스폰서, 백엔드/API 중심, 일반 방법론, 독립 블로그 홍보 글을 발행본에 남기지 않았는가.

리뷰 결과는 글마다 아래 형식으로 남깁니다.

```markdown
### 리뷰 결과: [번호] [플랫폼] 제목

- 판정: 통과 / 수정 필요 / 제외
- 원문 확인 URL:
- 원문에서 확인한 핵심 사실:
  - 
  - 
  - 
- 수정 필요:
  - 
- 카테고리 점검:
- 발행 전 조치:
```

`수정 필요`가 나오면 리뷰 에이전트는 바로 수정안을 만듭니다. 수정안은 원문에서 확인한 핵심 사실을 기준으로 제목, deck, 요약 bullet, 매거진 인사이트, 카테고리, 태그 중 필요한 부분을 고칩니다. 수정 후에는 같은 항목을 다시 리뷰해 `통과`로 바뀌었는지 확인합니다.

`제외`가 나오면 본문을 억지로 살리지 않습니다. 대신 해당 후보를 `수집했지만 제외한 것`으로 옮기고, 제외 사유와 다시 볼 조건을 남깁니다.

## 수정 루프

리뷰 에이전트는 아래 순서로 수정 루프를 수행합니다.

1. 원문을 다시 열고 핵심 사실 3개 이상을 추출합니다.
2. 작성본에서 원문과 다른 문장, 흐릿한 일반론, 빠진 기능명/도구명/API명을 표시합니다.
3. 제목, deck, 요약 bullet, 인사이트 본문을 원문 기준으로 다시 씁니다.
4. 카테고리와 태그가 틀리면 함께 수정합니다.
5. 수정한 글을 다시 리뷰해 `통과 / 수정 필요 / 제외` 중 하나로 판정합니다.
6. 두 번 수정해도 원문 기반 구체성이 생기지 않으면 `제외`로 돌립니다.

## 즉시 수정 조건

아래 조건 중 하나라도 발견되면 리뷰 에이전트는 발행을 멈추는 대신 수정 루프를 실행합니다.

- 출처 URL이 없거나 열리지 않습니다.
- 원문을 읽지 않고도 쓸 수 있는 일반론으로 본문이 구성되어 있습니다.
- 원문 핵심 도구/기능/API가 제목이나 요약에서 빠졌습니다.
- 대분류나 소분류가 잘못되어 잘못된 URL이 생깁니다.
- DEV 글인데 `구현 관점`, `실무 적용`, `같이 보면 좋은 기술`이 비어 있습니다.
- Service/Design 글인데 사용자 흐름, 화면, 제작/승인 기준이 보이지 않습니다.
- 원문에 없는 성과나 효과를 단정합니다.

## 6. 이미지/미디어 검수 에이전트

목표는 이미지가 글의 증거로 작동하는지 확인하는 것입니다.

검수 기준:

- 이미지가 최종 기준 원문, 공식 뉴스룸, 앱스토어/플레이스토어, 공식 블로그에서 온 것인지 확인합니다.
- 단순 OG 이미지나 분위기 이미지는 가능하면 쓰지 않고, UI 변화나 제품 상태를 설명하는 이미지를 우선합니다.
- 이미지 URL이 `https`로 접근 가능한지 확인합니다.
- 이미지가 없으면 임의 대체 이미지를 넣지 않고 `이미지 확보 메모`를 남깁니다.
- 출처가 후보 발견 글뿐인 이미지는 공식 원문 이미지로 대체하거나 비워둡니다.

## 7. 카테고리/라우팅 QA 에이전트

목표는 잘못된 URL과 소분류를 발행 전에 고치는 것입니다.

QA 기준:

- `dev/ai_design`, `dev/ai_dev`, `dev/data_api`처럼 역할이 섞인 경로를 찾습니다.
- 개발 AI는 `dev/ai`, 디자인 AI는 `design/ai`로 고칩니다.
- 화면 구현이나 접근성 QA와 연결되지 않는 DATA/API 글은 제외 후보로 돌립니다.
- `public/data/magazine.json`과 Notion 데이터의 `areaKey`, `categoryKey`, `area`, `category`가 같은 의미인지 확인합니다.
- article 번호와 route가 연속되는지 확인합니다.

## 8. Notion 동기화 에이전트

목표는 로컬 마크다운, JSON, Notion 데이터가 서로 다른 상태로 발행되는 일을 막는 것입니다.

동기화 기준:

- `runs/YYYY-MM-DD/magazine-report.md`, `public/data/magazine.json`, Notion DB의 제목, 플랫폼, 대분류, 소분류, deck, 태그를 비교합니다.
- 로컬에서 수정한 글은 Notion에도 같은 제목과 본문으로 반영합니다.
- Notion에서 삭제 또는 archive한 항목이 JSON에 다시 남아 있지 않은지 확인합니다.
- Notion 수정 후 다시 export했을 때 카테고리와 route가 유지되는지 확인합니다.

## 9. 뉴스레터 프리뷰 에이전트

목표는 매거진 글이 이메일에서도 읽히는지 확인하는 것입니다.

프리뷰 기준:

- 제목과 deck이 이메일 카드에서 너무 길게 보이지 않는지 확인합니다.
- 링크, 이미지, CTA가 깨지지 않았는지 확인합니다.
- 테스트 메일은 운영 규칙에 따라 먼저 지정된 테스트 수신자에게만 보냅니다.
- 테스트 확인 전에는 최종 수신자에게 발송하지 않습니다.
