#### 01. [당근] 동네 가게 후기 작성, 빈 입력창을 음성 인터뷰로 바꿨다

- 날짜: 2026-04-08
- 태그: AI후기 / 음성입력 / 로컬신뢰
- 국가: KR
- 카테고리: platform
- 직무 태그: 웹서비스기획 / 웹디자인
- 출처 유형: news
- 출처: 당근 보도자료
- 출처 URL: https://about.daangn.com/company/pr/archive/local-ai-review/
- 이미지: https://about.daangn.com/static/local-ai-review-hero.png
- 이미지 설명: 당근 동네지도 후기 작성 화면 공식 이미지
- 요약: 당근이 동네 가게 후기 작성 화면에 음성 입력 기반 AI 인터뷰 기능을 출시해, 방문 경험을 답하면 분위기·메뉴·서비스 특징까지 묻고 후기 초안으로 정리합니다. 1월 베타 기간 후기 품질 지표 40% 상승을 함께 공개했습니다.

##### 요약

- 당근은 후기 작성 화면을 빈 입력창에서 음성 답변 기반 인터뷰로 바꿔, 첫 한 줄을 떠올리지 못해 작성을 포기하던 흐름을 끊었습니다.
- AI는 분위기·서비스 특징·이용 메뉴를 차례로 묻고, 음성 답변을 모아 후기 초안으로 정리합니다.
- 사용자는 초안을 읽고 직접 수정한 뒤 등록하며, AI 보조 후기는 별도 표식으로 노출됩니다.
- 1월 베타에서 재이용률 43%, 후기 품질 지표 40% 상승이 공개됐고, 4월 전 업종으로 확대됐습니다.

##### 매거진 인사이트

> 빈 입력창을 없애는 AI는 리뷰 수를 늘리는 기능이 아니라, 이웃의 경험을 더 구체적인 지역 정보로 바꾸는 입력 UX입니다.

후기는 평점보다 늦게 쌓이는 정보입니다. 누군가 실제로 다녀온 장면, 메뉴, 응대, 분위기가 누적되어야 다음 방문자가 판단합니다. 당근은 이 지점을 빈 입력창이 아니라 대화형 인터뷰로 바꿨습니다.

###### 왜 지금 이 업데이트인가

로컬 서비스에서 후기 작성률은 검색·방문 결정·재방문에 직접 영향을 줍니다. 베타에서 재이용률과 품질 지표를 함께 공개한 것은 작성량보다 다시 쓰고 싶은 후기를 목표로 본다는 신호입니다. AI 리뷰 기능을 볼 때는 생성 문장의 자연스러움보다 사용자의 기억을 얼마나 구체적으로 꺼내는가가 먼저입니다.

###### 사용자는 무엇을 덜 해도 될까

사용자는 후기 제목, 평가 기준, 문장 구성을 처음부터 직접 정리하지 않습니다. 방문 경험을 말하면 서비스가 빠지기 쉬운 정보를 질문으로 보강합니다. 다만 최종 후기는 사용자가 확인하고 고쳐야 합니다.

###### 설계 관점

후기 작성 화면은 AI 질문, 사용자의 음성 답변, 생성된 초안, 최종 승인 단계를 분명히 나눠야 합니다. 질문은 "좋았나요?"보다 "무엇을 이용했고 어떤 장면이 기억났나요?"에 가까워야 후기가 추천 문구가 아니라 동네지도에서 다시 쓰일 수 있는 정보가 됩니다. AI 보조로 만들어졌는지 사용자가 수정했는지를 화면에서 구분해 신뢰 UX를 같이 설계합니다.

###### 점검 질문

- 우리 서비스의 리뷰 입력창은 사용자가 무엇을 써야 하는지 충분히 도와주고 있는가?
- AI가 만든 초안과 사용자가 직접 확인한 경험을 화면에서 구분할 수 있는가?
- 리뷰가 쌓이는 위치(검색 결과·지도·상세페이지)에 따라 질문 설계가 달라지고 있는가?

#### 02. [토스] 결제 영수증 카드에서 환불 가능 기간을 먼저 보여준다

- 날짜: 2026-04-22

- 국가: KR
- 카테고리: fintech
- 직무 태그: 웹서비스기획 / 웹디자인
- 출처 유형: release_note
- 출처: 토스 공지사항
- 출처 URL: https://www.toss.im/notice/payment-receipt-2026-04
- 이미지: https://static.toss.im/notice/payment-receipt-2026-04-hero.png
- 이미지 설명: 토스 거래내역 영수증 카드 개편 공식 이미지
- 요약: 토스가 거래내역 카드 상단에 환불 가능 기간과 결제 수단 변경 가능 여부를 노출하고, 카드를 펼치지 않아도 후속 행동을 결정할 수 있게 했습니다. 결제 수단 변경은 결제 후 7일 이내, 자동결제 등록 건만 적용됩니다.

##### 요약

- 토스는 거래내역 카드 상단을 결제 사실 표시에서 다음 행동 가능 여부 표시로 바꿔, 사용자가 카드를 펼치지 않아도 환불·변경 가능성을 확인할 수 있게 했습니다.
- 환불 가능 기간이 남은 거래는 카드 우측에 D-3 형식의 잔여 일수가 노출됩니다.
- 결제 수단 변경은 결제 후 7일 이내·자동결제 등록 건에만 적용되며, 즉시 결제 건은 변경이 아닌 환불 후 재결제로 안내됩니다.
- 영수증 카드 안에서 결제 수단 변경, 환불 신청, 사업자 정보 보기로 이어지는 후속 행동 동선이 한 화면에 정리됐습니다.

##### 매거진 인사이트

> 영수증은 결제 사실을 알리는 기록이 아니라, 사용자가 다음 행동을 결정하는 화면입니다.

토스는 거래내역을 끝난 화면이 아니라 결정 화면으로 봤습니다. 환불 가능 잔여 기간과 결제 수단 변경 조건은 보통 안내 페이지나 고객센터에서 확인하지만, 이번 개편은 그 정보를 카드 첫 줄로 끌어올렸습니다.

###### 왜 지금 이 업데이트인가

결제 후 사용자가 가장 자주 묻는 것은 환불 가능 여부와 결제 수단 변경 가능 여부입니다. 이 정보를 안내 페이지 깊은 곳에 두면 고객센터 문의로 이어지고, 운영 비용도 같이 늘어납니다.

###### 사용자는 무엇을 덜 해도 될까

사용자는 결제 후 환불 가능한지, 카드 변경이 되는지 검색하거나 문의하지 않아도 됩니다. 카드 첫 줄에서 잔여 일수와 변경 가능 조건이 보이면, 다음 행동을 즉시 시작할 수 있습니다.

###### 설계 관점

영수증 카드의 첫 줄은 결제 결과보다 결정 가능성이 우선합니다. 잔여 일수, 변경 가능 조건, 후속 행동 버튼이 같은 시야 안에 있어야 합니다. 변경 불가 케이스는 사유와 대안 행동(환불 후 재결제)을 흐리지 말고 명시합니다.

###### 점검 질문

- 우리 서비스의 거래내역은 결제 사실 표시에 머무는가, 다음 행동을 결정하는 화면인가?
- 환불·취소·교환 가능 조건이 화면 첫 줄에서 보이는가, 안내 페이지를 열어야 보이는가?
- 변경 불가 케이스의 대안 행동이 사용자가 한 번에 시작할 수 있게 노출되는가?

- 출처 유형: release_note
- 출처: 쿠팡 뉴스룸
- 출처 URL: https://news.coupang.com/2026/04/price-panel/
- 이미지: https://newsimg.coupangcdn.com/2026/04/price-panel-hero.png
- 이미지 설명: 쿠팡 상품 상세 가격 패널 공식 이미지
- 요약: 쿠팡이 상품 상세에서 일반 가격·로켓와우 가격·묶음 할인·배송비를 별도 안내 영역에 흩어 두지 않고, 결제 직전 한 패널에 모아 보여주도록 개편했습니다. 같은 SKU의 묶음 할인 도달 여부도 상품 상세에서 미리 확인할 수 있습니다.

##### 요약

- 쿠팡은 상품 상세의 가격 영역을 일반가·멤버십가·묶음할인·배송비가 한 패널에 정렬되는 구조로 바꿨습니다.
- 묶음 할인 도달까지 남은 수량은 가격 패널 하단에 "1개 더 담으면 5,000원 추가 할인"처럼 잔여 조건 형식으로 표시됩니다.
- 로켓와우 가격은 미가입자에게도 노출되지만, 결제 단계에서 가입 후 적용 안내가 분리되어 보입니다.
- 같은 패널 안에서 정기배송 가격과 일회성 구매 가격이 토글로 비교되어, 카트로 보내기 전에 비교가 끝납니다.

##### 매거진 인사이트

> 상품 상세는 정보를 모아두는 페이지가 아니라, 결제 직전 의사 결정을 압축하는 화면입니다.

쿠팡은 가격 정보가 흩어져 있어 사용자가 카트와 결제 화면을 오가며 다시 계산하던 흐름을 줄였습니다. 한 패널에 묶음·멤버십·배송비를 모은 것은 단순 정렬이 아니라 결제 직전 비교를 상품 상세 안에서 끝내겠다는 결정입니다.

###### 왜 지금 이 업데이트인가

이커머스에서 결제 직전 이탈은 가격 계산 부담에서 자주 발생합니다. 멤버십 가격, 묶음 할인, 배송비가 다른 영역에 있을수록 사용자는 카트, 결제, 다시 상품 상세를 오갑니다.

###### 사용자는 무엇을 덜 해도 될까

사용자는 카트에서 가격을 다시 계산하지 않아도 됩니다. 상품 상세에서 멤버십 적용가, 묶음 도달 조건, 배송비 합산을 한 번에 비교한 뒤 의사를 결정할 수 있습니다.

###### 설계 관점

가격 패널은 단순 가격 표시 영역이 아니라 결제 직전 의사 결정 모듈입니다. 멤버십 미가입자에게는 가격 노출과 가입 안내를 분리하고, 묶음 할인 잔여 조건은 행동을 유도하는 문구로 작성해야 합니다. 정기·일회성 토글은 카트로 옮기지 말고 상세 안에서 비교가 끝나야 합니다.

###### 점검 질문

- 우리 상품 상세는 가격·할인·배송비가 한 시야에서 비교 가능한가?
- 묶음 할인이나 임계 조건이 잔여 조건 형식으로 사용자 행동을 유도하는가?
- 멤버십 가격 노출이 미가입자에게 혼란이 되지 않도록 가입 안내와 분리되어 있는가?

- 요약: 무신사가 스타일 컬렉션 카드를 코디·아이템·발매 일정이 한 카드 안에서 이어지는 구조로 개편해, 룩북에서 발견한 코디를 그대로 저장하고 미발매 상품의 발매 알림까지 같은 카드에서 받을 수 있게 했습니다.

##### 요약

- 무신사는 스타일 컬렉션 카드를 룩북 이미지·아이템 목록·발매 일정이 한 카드에서 이어지는 구조로 개편했습니다.
- 코디 저장은 사용자별 폴더로 분리되며, 저장한 코디 안의 미발매 상품에는 발매 알림이 자동으로 따라붙습니다.
- 같은 코디를 다른 색상·사이즈로 바꿔보는 버튼이 카드 안에 들어와, 상세 페이지로 이동하지 않고도 비교가 됩니다.
- 발매 후에는 카드 상태가 "발매 완료"로 자동 갱신되고, 같은 코디의 재구매·후기·룩북 재등장 정보가 이어 노출됩니다.

##### 매거진 인사이트

> 패션 커머스의 룩북은 보고 끝나는 콘텐츠가 아니라, 발견·저장·재방문을 한 카드 안에서 연결해야 하는 입구입니다.

- ECMAScript Iterator Helpers 제안이 stage 4에 진입해 표준 명세에 포함됐고, V8·SpiderMonkey·JavaScriptCore 모두 구현 단계에 들어갔습니다.

> Iterator Helpers는 단순 편의 메서드 추가가 아니라, JavaScript에서 lazy 컬렉션 처리를 라이브러리 없이 다룰 수 있게 만드는 변화입니다.

###### 구현 관점

기존 Array 메서드 chain은 즉시 평가되지만 Iterator Helpers는 lazy입니다. 같은 코드처럼 보여도 평가 시점이 다르므로, 디버깅과 단위 테스트 작성 시점에 평가 트리거를 명확히 해야 합니다.

###### 실무에 어떻게 적용할 수 있을까

- API 응답을 streaming JSON으로 받는 경우, ReadableStream을 iterator로 변환해 `map`·`filter`로 직접 처리합니다.
- IndexedDB cursor 처리에서 `take`·`drop`을 사용하면 페이지 단위 조회를 임시 배열 없이 구현할 수 있습니다.
- generator 기반 데이터 처리 코드는 외부 라이브러리 의존성을 정리하기 좋은 시점입니다.

###### 같이 보면 좋은 기술

- AsyncIterator Helpers: 비동기 iterator에 같은 메서드를 추가하는 후속 제안. stage 3.
- ReadableStream.from(): iterator를 stream으로 변환하는 표준화 흐름과 함께 보면 좋습니다.

#### 20. [Mozilla] Firefox 150이 ariaNotify()를 지원해 라이브 리전 한계를 보완한다

- 날짜: 2026-04-22
- 태그: ariaNotify() / live region / Firefox 150
- 국가: GLOBAL
- 카테고리: Accessibility

- 출처: MDN
- 출처 URL: https://developer.mozilla.org/en-US/docs/Web/API/Element/ariaNotify
- 이미지:
- 이미지 설명: 이미지 확인 필요
- 요약: Firefox 150이 `Document`와 `Element`의 `ariaNotify()`를 지원해 DOM 변경에 묶이지 않은 스크린리더 알림을 queue에 넣을 수 있게 했고, 기존 ARIA live region이 처리하기 어려웠던 비동기 알림 시점 문제를 보완합니다.

##### 용어 설명

- ARIA live region: DOM 영역에 `aria-live` 속성을 두고 그 안의 텍스트가 변경되면 보조기술이 자동으로 읽어주는 패턴입니다. 변경 시점이 DOM 업데이트와 묶여 제어가 까다롭습니다.

##### 요약

- Firefox 150이 `Document.ariaNotify()`와 `Element.ariaNotify()`를 지원해, DOM 변경 없이 스크린리더에 알림 메시지를 보낼 수 있게 됐습니다.
- 메시지는 우선순위 옵션과 함께 queue에 들어가며, live region의 race condition 없이 순서가 유지됩니다.
- 기존 `aria-live="polite"` 패턴이 다루기 어려웠던 비동기 작업 완료 알림(예: 파일 업로드 진행 상황) 처리에 적합합니다.
- Chrome·Safari는 아직 미지원이므로 progressive enhancement 패턴으로 적용해야 합니다.

##### 매거진 인사이트

> `ariaNotify()`는 ARIA live region의 종료 신호가 아니라, 비동기 알림 시점 제어를 명시적으로 만드는 보완 도구입니다.

라이브 리전은 DOM 변경에 묶여 있어 알림 순서나 시점이 보조기술 구현마다 달라지는 문제가 있었습니다. `ariaNotify()`는 이 문제를 명시적 호출 방식으로 해결합니다.

###### 왜 지금 이 업데이트인가

비동기 작업이 많은 SPA에서 라이브 리전은 메시지 누락이나 중복 발화 문제를 종종 일으킵니다. `ariaNotify()`는 이 케이스에 직접 대응하는 표준 API로, 접근성 QA 항목에 새 패턴을 추가합니다.

###### 구현 관점

`ariaNotify()`는 Firefox 150에만 있으므로 기존 라이브 리전 패턴을 바로 제거하지 말고 `if (element.ariaNotify)` 분기로 감싸야 합니다. queue 우선순위는 `important`/`none`으로 분리되며, 알림이 너무 많으면 사용자에게 부담이 되므로 빈도 제한을 함께 둡니다.

###### 실무에 어떻게 적용할 수 있을까

- 파일 업로드·결제·검색 결과 갱신처럼 비동기 완료 알림이 중요한 흐름에 progressive enhancement로 도입합니다.
- 기존 `aria-live` 영역은 유지하고, 미지원 브라우저에서는 동일 메시지가 fallback으로 발화되도록 합니다.
- 접근성 QA에서는 알림 발화 순서·간격·중복 여부를 NVDA·VoiceOver·TalkBack에서 각각 확인합니다.

###### 같이 보면 좋은 기술

- ARIA Live Regions: 같은 영역의 기존 패턴. 단순 텍스트 변경에는 여전히 가장 호환성 높은 선택입니다.
- Notifications API: OS 수준 알림은 `ariaNotify()`와 다른 채널입니다. 혼동하지 않도록 분리합니다.

- catalog: workspace의 여러 패키지가 같은 dep을 같은 버전으로 쓰도록 한 곳에 정의하는 pnpm의 새로운 기능입니다.

##### 요약

- pnpm 9가 정식 출시되어 monorepo workspace의 dep hoisting 모델을 더 엄격하게 바꿨고, undeclared dep을 자동 hoist하던 동작이 기본 비활성화됐습니다.
- `catalog` 기능이 안정화되어, workspace 안 여러 패키지가 같은 dep을 같은 버전으로 사용하도록 한 곳에서 정의할 수 있습니다.
- `pnpm.overrides` 동작이 명세 기반으로 정리되어, 한 모노레포 안에서 transitive dep을 일괄 고정하기 쉬워졌습니다.
- 기존 v8 프로젝트는 lockfile 마이그레이션이 자동이며, 일부 strict 옵션은 `pnpm-workspace.yaml`에서 점진 적용 가능합니다.

##### 매거진 인사이트

> 패키지 매니저의 다음 발전은 설치 속도가 아니라, 모노레포가 커졌을 때의 의존성 일관성입니다.

대형 monorepo에서는 같은 dep의 여러 버전이 섞이는 문제가 가장 자주 깨지는 지점입니다. pnpm 9의 `catalog`는 이 문제를 명세 수준에서 해결합니다.

###### 왜 지금 이 업데이트인가

monorepo 채택이 늘면서 의존성 일관성과 보안 패치 속도가 중요해졌습니다. catalog와 strict hoisting은 이 두 축을 동시에 강화합니다.

###### 구현 관점

`pnpm.catalog`는 `pnpm-workspace.yaml`에 정의되며, 패키지 `package.json`에서는 `"react": "catalog:"` 형태로 참조합니다. strict hoisting은 기존 코드에서 undeclared import가 있던 경우 빌드가 깨질 수 있으므로 점진 적용 옵션을 활용합니다.

###### 실무에 어떻게 적용할 수 있을까

- 모노레포에서 React·React DOM·디자인 시스템 패키지처럼 버전 일치가 필수인 dep은 `catalog:`로 옮깁니다.
- strict hoisting 도입 전에 `pnpm install --strict-peer-dependencies` 같은 dry run으로 영향 범위를 확인합니다.
- 보안 패치는 `pnpm.overrides`로 transitive dep 버전을 일괄 고정해 즉시 배포합니다.

###### 같이 보면 좋은 기술

- Yarn Berry constraints: 같은 문제 영역(모노레포 일관성)을 다른 방식으로 해결합니다. 비교 시점에 함께 봅니다.
- Renovate / Dependabot: catalog 사용 시 자동 PR 도구가 catalog 인식하도록 설정해야 합니다.
