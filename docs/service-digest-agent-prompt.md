# Service Digest Agent Prompt

이 문서는 `npm run fetch:service`로 모은 `runs/YYYY-MM-DD/service-articles.json`을 읽고, 웹서비스기획자와 PM이 바로 회의에 가져갈 만한 SERVICE 후보를 고르는 기준입니다.

수집 스크립트는 후보와 단서 태그만 모읍니다. LLM API를 호출하지 않고, 최종 채택/보류/제외 판단과 원문 확인, 요약 작성은 작업 AI가 이 문서를 기준으로 수행합니다.

## 목적

SERVICE는 "이커머스 산업 중심 인사이트"를 단일 축으로 운영합니다. 좋은 기사 몇 건을 고르는 큐레이션이 아니라 `CTTD 클라이언트 관심사`를 매주 모니터링하고 그중 무엇을 남겼는지 판단하는 운영 작업으로 봅니다.

이커머스 5축 (메인 채택 기준):

1. 상품 탐색·홈 — 홈 개편, 카테고리 분리, 개인화 피드, 브랜드관, 큐레이션, 검색 결과 UI
2. 비교·선택 — AI 추천, 유사 상품, 듀프, 리뷰 요약, 성분/취향/사이즈 기반 비교, 필터 개선
3. 구매·전환 — 장바구니, 결제, 쿠폰, 포인트, 멤버십, 페이백, 구독, 무료배송 조건
4. 멤버십·재방문 — 멤버십 등급, 알림, 찜, 재입고, 출석, 리워드, 앱 전용 혜택, CRM
5. 리뷰·추천·O2O — 리뷰 UX, AI 후기, 추천 근거 노출, 매장 픽업, 앱 쿠폰 회수, 방문 후 CRM

Tier 구조:

- 1급 관심사: 패션 커머스, 뷰티 커머스, 종합 커머스/마켓플레이스, 멤버십·CRM, 프로모션형 구매 경험. 매주 기본 모니터링 대상.
- 2급 참고군: 브랜드 콘텐츠 커머스, 생활/가전 커머스, 로컬 서비스, 핀테크, 리서치/조사 자료. 클라이언트 회의에서 바로 질문으로 바꿀 수 있을 때만 메인 후보.

각 회차는 먼저 어떤 관심사 축과 플랫폼을 모니터링했는지 정리하고, 그다음 최종 채택 항목과 제외 항목을 남깁니다.

좋은 후보는 이커머스 5축 중 하나에서 화면·플로우·정책·조건·추천 근거·운영 흐름이 구체적으로 바뀐 것을 보여줘야 합니다.

명시 제외:

- 글로벌 비이커머스 콘텐츠 서비스(예: Spotify, Netflix 같은 음악·영상 콘텐츠 자체) — SERVICE 스코프 밖
- 제휴카드, e쿠폰, 단순 프로모션, 오프라인 이벤트, 비즈니스 실적, 채용, ESG, 브랜드 캠페인 (원문에 구체 화면·플로우·정책 변화가 없으면 제외)
- 단순 매출/거래액/투자 소식 (사용자 화면이 안 바뀌면 제외)

유명 브랜드 소식이어도 화면, 플로우, 정책, 조건 노출, 추천 근거, 검수/승인, 재방문 트리거가 확인되지 않으면 메인 글로 만들지 않습니다.

## 입력

- `runs/YYYY-MM-DD/service-articles.json`
- 필요 시 원문 URL
- 필요 시 후보 발견 출처가 가리키는 공식 발표, 제품 공지, 뉴스룸, 앱스토어/플레이스토어, 보도자료, 리서치 원문

## 출력

`shortlist-20-30.md` 또는 `magazine-report.md`에 넣기 전에 각 후보마다 아래 판단을 남깁니다.

- 모니터링 축: 어느 CTTD 관심사 축에서 본 후보인가
- 판정: `채택 / 보류 / 제외`
- 타겟 판정: `core_ecommerce / commerce_adjacent / weak_promo / exclude`
- 우선순위: `P0 / P1 / P2 / 제외`
- 근거 화면: 홈 / 검색 / 상품상세 / 장바구니 / 결제 / 마이페이지 / 리뷰 / 추천 / 멤버십 / 운영도구 / 없음
- 직무 태그: `웹서비스기획`
- 출처 유형: `news / release_note / blog_opinion / reference / guide / research`
- 태그: [공통 태그 원칙](editorial-style-guide.md#공통-태그-원칙)에 따라 작성
- 고객 회의 질문: 우리 서비스에 바로 물어볼 수 있는 한 문장
- 원문 확인 URL
- 제외 사유: 제외일 때만 작성

## 채택 기준

아래 3문을 모두 통과한 뒤, 세부 기준 중 하나 이상이 원문에서 구체적으로 확인되면 채택 후보입니다.

1. CTTD 클라이언트 적합성
   패션, 뷰티, 종합 커머스/마켓플레이스, 멤버십·CRM, 프로모션형 구매 경험 중 어디에 직접 닿는지 설명할 수 있어야 합니다. 2급 참고군은 고객 회의에서 바로 꺼낼 수 있을 때만 남깁니다.

2. UIUX 변화의 구체성
   화면, 플로우, 정책, 조건, 추천 근거, 운영 흐름 중 실제로 무엇이 바뀌었는지 원문 기준으로 설명할 수 있어야 합니다.

3. 미팅 전환 가능성
   "우리 서비스에서는 무엇을 점검해야 하지?"라는 질문을 한 문장으로 바꿀 수 있어야 합니다.

그다음 아래 중 하나 이상이 원문에서 구체적으로 확인되면 채택 후보입니다.

1. 이커머스 핵심 흐름
   상품 탐색, 추천, 상세 정보, 리뷰, 장바구니, 결제, 배송, 픽업, 멤버십, 재구매 흐름 중 하나가 실제로 바뀐다.

2. 정책과 조건 노출
   혜택 조건, 수수료, 검수, 승인, 환불, 예약 가능 조건, 멤버십 등급처럼 사용자가 판단해야 하는 조건이 화면이나 정책으로 드러난다.

3. 신뢰와 전환
   리뷰, 후기, 판매자 정보, 프로필, 인증, 안전거래, 추천 근거처럼 사용자의 불안을 줄이거나 다음 행동을 결정하게 하는 장치가 있다.

4. 운영자 UX
   판매자, 운영자, 상담자, 관리자 업무에서 검색, 확인, 승인, 조치, 정산, 광고 운영, 데이터 조회 흐름이 바뀐다.

5. 리서치 근거
   소비자 조사, 서비스 이용 행태, 카테고리 변화가 구체 수치와 조건을 갖추고 있고, 웹서비스 화면·정책 질문으로 바꿀 수 있다.

## 제외 기준

아래 후보는 원칙적으로 제외합니다.

- 제휴카드, 쿠폰, e쿠폰, 멤버스데이, 단순 할인전처럼 프로모션 사실만 있는 글
- 콘텐츠 제휴, 브랜드 협업, 캠페인 공개처럼 화면·플로우·정책 변화가 없는 글
- 오프라인 매장, 팝업, 전시, 행사 소식만 있고 앱 예약, 쿠폰 회수, 픽업, 멤버십 연동, 방문 후 CRM 흐름이 없는 글
- 투자, 실적, 조직, 채용, ESG, 사회공헌, 수상 소식
- 외부 AI 연동이지만 계정 연결, 권한, 추천 결과, 전환 화면, 운영자 조치 흐름이 확인되지 않는 글
- 원문을 읽지 않고도 쓸 수 있는 "혜택을 쉽게 보여준다", "다음 거래로 연결한다" 같은 일반론만 남는 글

## 단서 태그 사용법

`service-articles.json`의 `evidenceTags`와 `riskTags`는 자동 판정이 아니라 검토 단서입니다.

- `evidenceTags`가 있어도 원문에서 실제 화면·정책·플로우 증거가 없으면 제외합니다.
- `riskTags`에 `weak_promo`, `partnership_only`, `offline_only`, `business_only`가 있으면 먼저 제외 가능성을 검토합니다.
- `sourceRole`이 `discovery`인 후보는 후보 발견 출처입니다. 공식 원문이나 신뢰 가능한 2순위 출처를 찾지 못하면 메인 글로 올리지 않습니다.

## 출력 기준 연결

SERVICE 글의 최종 출력 형식은 [매거진 원고 출력 기준](magazine-writing-standard.md#service-출력-형식)을 따릅니다.

각 SERVICE 항목은 메타데이터에 `출처 유형`을 반드시 작성합니다. 공식 발표나 보도자료는 `news`, 공식 제품 업데이트는 `release_note`, 리서치나 조사 자료는 `research`, 방법론 글은 `guide`, 관점 중심 블로그는 `blog_opinion`, 서비스 사례 참고 글은 `reference`를 사용합니다.

요약 메타와 본문 요약은 [공통 요약 원칙](editorial-style-guide.md#공통-요약-원칙)을 따릅니다.

태그는 [공통 태그 원칙](editorial-style-guide.md#공통-태그-원칙)을 따릅니다.

SERVICE 본문은 `요약`과 `매거진 인사이트`를 유지합니다. `요약`은 원문 전체 핵심을 불릿 리스트로 정리하는 본문 섹션입니다. 세부 라벨과 작성 기준은 [매거진 원고 출력 기준](magazine-writing-standard.md)을 단일 기준으로 봅니다.

`매거진 인사이트` 소제목은 4개 잠금 셋 외 추가하지 않습니다: `왜 지금 이 업데이트인가` / `사용자는 무엇을 덜 해도 될까` / `설계 관점` / `점검 질문`. 다른 변형 소제목은 표준 문서에서 폐기했습니다.

요약은 자체 완결되어야 합니다. 원문을 열지 않은 독자가 메타 `요약:`과 본문 `##### 요약`의 첫 불릿(글 전체 결론 한 줄)만 읽어도 누가 무엇을 했고 무엇이 핵심 변화이며 왜 봐야 하는지 그려져야 합니다. `~를 다룹니다`, `~를 소개합니다` 같은 표지 문장으로 시작하지 않습니다. 본문 요약 불릿 개수는 제한하지 않으며, 원문 밀도에 맞게 필요한 만큼 담습니다.

SERVICE 회차 운영 메모에는 아래 3층이 보여야 합니다.

- 이번 주 모니터링 범위: 어떤 관심사 축과 플랫폼을 봤는지
- 채택한 TOP 항목: 무엇이 바뀌었고 왜 남겼는지
- 수집했지만 제외한 것: 왜 뺐고 다시 볼 조건이 무엇인지

권장 회차 템플릿:

```markdown
## Service

### 이번 주 모니터링 범위

- 중점 관심사 축:
- 확인 플랫폼:
- 이번 주 포인트:

### 채택한 TOP 항목

#### 01. ...

### 수집했지만 제외한 것

- 플랫폼/이슈명
- 제외 이유
- 다시 볼 조건
```

## 원문 확인 규칙

- 공식 공지, 뉴스룸, 보도자료, 앱스토어/플레이스토어, 리서치 원문을 우선합니다.
- 큐레이션, 뉴스레터, 개인 블로그에서 발견한 후보는 글 안의 링크, 제품명, 기능명, 날짜를 따라가 최종 기준 원문을 찾습니다.
- 원문에서 발행에 쓸 수 있는 구체 사실 3개 이상을 추출하지 못하면 제외합니다.
- 이미지는 최종 기준 원문, 공식 뉴스룸, 앱스토어/플레이스토어, 공식 블로그에서 제공한 대표 이미지만 사용합니다.
